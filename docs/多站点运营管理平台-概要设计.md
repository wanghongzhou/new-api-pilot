# 多站点运营管理平台 — 概要设计

> 文档状态：开发基线版（本期范围内需求已闭环，无待确认项）  
> 详细设计索引：[多站点运营管理平台-详细设计.md](./多站点运营管理平台-详细设计.md)

## 1. 建设目标

公司运营多个独立部署的 new-api 站点，需要建设一个独立的内部运营平台，通过各站点公开或管理 HTTP API 完成：

1. 站点接入、授权、在线状态和实例资源监控；
2. 指定远端用户的纳管及历史用量统计；
3. 客户与纳管账户的固定归属管理；
4. 账户、客户、站点、全局、模型、通道六个层级的统计；
5. Dashboard、详细统计、Excel/CSV 导出；
6. 数据缺口识别、历史回填、次日校验和周期校验；
7. 站点、采集、实例资源和纳管账户异常告警。

平台只负责运营统计和监控，不参与 AI 请求转发。平台故障不得影响任何 new-api 站点的正常服务。

### 1.1 使用角色与职责

| 角色 | 主要目标 | 本期责任边界 |
|---|---|---|
| 平台 admin | 接入站点、维护客户和账户、处理采集异常、配置告警与系统参数 | 对站点身份、生产接入清单、危险操作和回填结果负责 |
| 平台 viewer | 查看、筛选、下钻和导出运营数据 | 不接触远端凭据，不执行任何写操作 |
| 运维人员 | 部署、监控、备份、恢复和密钥保管 | 不通过产品页面修改业务统计，不直接手工改统计表 |
| 测试/验收人员 | 按需求追踪和验收矩阵验证功能、契约与非功能指标 | 保存可复现 fixture、命令输出和验收证据 |

### 1.2 开发就绪与上线成功标准

设计达到开发就绪必须同时满足：本期每项需求均能追踪到详细设计、API/任务、页面或运维动作以及验收编号；所有状态都有进入、可见结果、失败处理和恢复路径；所有页面所需字段均存在于权威 DTO；本期未决需求数为 0。该结论只表示可以直接开发，不表示实现已经完成。

实现达到上线标准必须满足：核心验收矩阵全部通过，固定容量画像满足性能目标，生产接入清单逐站签署，备份恢复演练满足 RPO/RTO，发布门禁无豁免项。任何失败项都会重新打开对应需求，不允许用“内部项目”作为跳过数据正确性、安全或恢复验证的理由。

## 2. 系统边界

### 2.1 独立部署

本平台是独立产品，不作为当前 new-api 服务的功能模块运行：

- 独立 Go 后端进程；
- 独立 React 前端；
- 独立 MySQL 数据库；
- 独立 Redis，用于快速周期任务的有界短期历史，不承担业务事实、分布式锁或持久任务恢复；
- 独立配置、日志、迁移、调度器和发布流程；
- 不向 new-api 当前 Router、Controller、Model 或 `web/default` 直接加入平台路由和业务表；
- 当前 new-api 源码仅作为分层架构、API 契约和前端 default 视觉/交互风格的参考实现。

平台后端继续采用 Router → Controller → Service → Model 分层，前端参考 `web/default` 的 React、Base UI、Tailwind、TanStack Router、TanStack Query 和 VChart 使用方式，但代码位于平台自身工程中。

详细设计：[总体架构与安全](./多站点运营管理平台-详细设计-01-总体架构与安全.md)、[后端实现](./多站点运营管理平台-详细设计-05-后端实现.md)、[前端实现](./多站点运营管理平台-详细设计-06-前端实现.md)。

### 2.2 本期范围

- 六级用量统计及小时、日、月、年查询；
- CPU、内存、磁盘、实例在线状态和当前 RPM/TPM；
- 站点性能健康实时摘要，以及站点列表使用的进程内短期缓存；
- probe、RPM/TPM、资源快照快速任务的 Redis 有界执行历史；
- 数据回填、缺口修复和一致性校验；
- Excel、CSV 后台导出；
- 钉钉机器人告警；
- admin、viewer 两级平台权限；
- 前端仅支持简体中文 `zh-CN`，不提供语言检测或切换；所有用户可见文案仍统一经 i18next 管理；
- 单实例平台部署；
- 仅支持 MySQL 8.0 及以上版本；开发与验收基线使用 MySQL 8.4。
- 接入站点由内部团队统一维护并保持协议兼容；平台不做版本号锁定，升级前应验证 `/api/status`、数据导出及身份接口契约。

### 2.3 本期不做

- 财务对账、账单、欠费、付款、发票和调账；
- 远端用户创建、充值、调额和禁用，以及 new-api 用户 API Token 管理；
- 财务结算与对账；通道和站点运营数据仅提供只读采集、统计、完整性和告警展示；
- SSO、2FA 和复杂密码策略；
- 多实例平台高可用和分布式调度；
- 模型别名合并；
- 客户合并、账户跨客户迁移和历史归属变更。

quota 和换算金额只用于运营展示，不作为财务结算依据。

## 3. 总体架构

~~~text
┌──────────────────────────────────────────────────────┐
│                多站点运营管理平台                    │
│                                                      │
│  React Web ── Go API ── Scheduler/Worker ── Alert    │
│                    ├── MySQL（事实、任务、告警）     │
│                    └── Redis（快速任务短期历史）     │
└───────────────┬────────────┬────────────┬────────────┘
                │ HTTP API   │            │
          ┌─────▼─────┐ ┌────▼─────┐ ┌────▼─────┐
          │ new-api 1 │ │ new-api 2│ │ new-api N│
          └───────────┘ └──────────┘ └──────────┘
~~~

用量、Dashboard、资源历史和普通业务页面查询只读取平台 MySQL，不在单次请求中串行穿透多个站点。明确例外是站点详情性能健康：`GET /api/sites/:id/performance` 对单站实时代理上游性能摘要；站点列表不阻塞等待上游，而是读取按 `site_id + config_version` 隔离的进程内 24 小时摘要缓存，缓存缺失时异步刷新。快速任务历史从 Redis 读取，Redis 不保存业务事实。

## 4. 核心业务对象

| 对象 | 定义 |
|---|---|
| Site | 一套独立部署的 new-api 服务 |
| Remote User | 某个站点中的 new-api 用户 |
| Account | 平台纳管的统计账户，固定绑定一个站点远端用户和一个客户 |
| Customer | 平台业务客户，可拥有多个纳管账户 |
| Channel | 站点中的上游通道，仅同步 ID 和名称用于统计展示 |
| Instance | new-api 运行实例，以站点内的 node_name 标识 |

账户的 site_id、remote_user_id、customer_id 创建后不可修改。本期不支持账户转移客户；如未来需要转移，必须另行设计带生效时间的历史归属模型。

详细设计：[数据模型](./多站点运营管理平台-详细设计-03-数据模型.md)。

## 5. 统计范围和事实源

| 层级 | 数据范围 |
|---|---|
| 账户 | 单个纳管账户 |
| 客户 | 客户名下纳管账户 |
| 站点 | 单站全部远端用户 |
| 全局 | 所有纳入统计的站点全部远端用户 |
| 模型 | 单站全部用户，按原始 model_name |
| 通道 | 单站全部用户，按 channel_id |

`usage_fact_hourly` 是唯一事实源，粒度为：

~~~text
(site_id, remote_user_id, model_name, channel_id, hour_ts)
~~~

所有小时窗口采用整站整小时替换；所有汇总必须能够从事实表重建。稀疏存储通过 `collection_window` 区分完整零流量和未知缺口：

- complete 且没有事实记录：返回 0；
- partial：返回已知合计并标记不完整；
- missing、pending、unavailable、paused：不得伪装为 0。

性能健康不是 `usage_fact_hourly` 的派生统计，也不是历史事实。详情按所选 1～720 小时实时读取上游滚动摘要；只有 24 小时结果进入单实例进程内缓存，成功缓存 1 小时、失败状态缓存 5 分钟，进程重启后丢失并按需重建。不得把滚动摘要相加或把缓存描述为数据库历史。

详细设计：[数据采集与统计](./多站点运营管理平台-详细设计-02-数据采集与统计.md)。

## 6. 历史范围

每个站点必须具有明确且对齐北京时间整点的 `statistics_start_at`。本项目约定首个用户必为不可删除的 root 用户，因此站点接入采用“创建基础信息 → 授权身份校验 → 证明首个 root 身份并自动确定历史起点 → 能力检查 → 同步 → 历史回填”的固定顺序：

1. 授权后通过 `GET /api/user/:root_user_id` 精确读取候选 root 用户；
2. 必须校验该用户 role=100、status=1、0<created_at<=当前时间；
3. 使用同一授权读取一次完整、分页一致的远端用户快照，要求候选 root 的 ID 是快照中的最小正 ID，且其 created_at 不晚于快照内任何用户；无法完成完整快照或证明首用户身份时按 SITE_INCOMPATIBLE 拒绝接入；
4. `statistics_start_at = floorToBeijingHour(root.created_at)`，`statistics_start_source=root_created_at`；
5. 起点一经首次写入即不可在产品中修改，重新授权也必须重复首用户证明且不得重算或覆盖；
6. 无法读取合法首个 root 用户时站点接入失败并标记 SITE_INCOMPATIBLE，不允许仅凭 role=100 或人工声明猜测历史起点；
7. 起点之前的数据定义为平台统计范围外，不计入完整率。

完整历史还依赖运营不变量：站点从首笔业务请求前就已开启 enable_data_export，且 new-api 的 quota_data 未被清理。当前上游接口无法证明历史开关时间；不满足该前提的站点不在“完整历史”承诺范围内，接入前必须由部署清单确认。

“首个且不可删除”中的“首个”由完整用户快照的最小正 ID 和最早 created_at 双重校验；“不可删除”依赖受支持 new-api 版本对 root 角色的权限约束，并由版本基线和接入清单共同保证。若未来上游不再提供上述可验证语义，平台必须先修订历史起点方案，不能降级为人工填写。

历史回填窗口也必须执行 flow/data 一致性校验；不能只校验最近一天或最近七天。

## 7. 站点授权

平台提供两种授权方式：

### 7.1 录入已有 Access Token，推荐

admin 输入 root_user_id 和已存在的 root Access Token。平台只保存加密后的 Token，并通过只读管理接口校验权限和能力，不修改远端 Token。

### 7.2 登录并重新生成 Access Token，可选

admin 输入 root 用户名和密码，平台登录后调用 new-api `GET /api/user/token`。当前 new-api 实现会生成新的 Access Token 并写回用户记录，因此该操作会覆盖原 root Access Token，可能使其他依赖原 Token 的系统失效。

该方式必须：

- 在 UI 中明确展示覆盖影响；
- 要求 admin 二次确认；
- 密码只在本次请求内使用并立即释放；
- 授权结果不返回或记录完整 Token；
- 失败时不得保存密码或半成品凭据。

生产环境优先录入站点首个且不可删除的 root 用户现有 Access Token；不得新建或改用后建的 root/admin 账户作为采集身份，否则其 created_at 不能代表站点历史起点。

详细设计：[总体架构与安全](./多站点运营管理平台-详细设计-01-总体架构与安全.md)。

## 8. 核心功能闭环

| 事件 | 系统处理 | 失败恢复 |
|---|---|---|
| 新建站点 | 录入→授权身份校验→读取 root.created_at→自动设置历史起点→能力检查→同步→回填 | 自动重试、手动补采、告警 |
| 小时采集 | 拉取 flow→校验→整小时替换→重建汇总 | 保留旧事实、标记缺口、重试 |
| 历史回填 | 按小时拉取 flow/data 并校验 | 失败窗口独立重试 |
| 新建账户 | 固定绑定→从本地事实重建历史 | 依赖缺失窗口补采后继续 |
| 站点/客户/账户暂停 | 停止相应采集或汇总 | 恢复后按暂停边界补齐 |
| 数据不一致 | 标记 missing→保留旧值→告警 | 重新拉取并整体替换 |
| 资源异常 | 连续评估→firing→钉钉 | 自动恢复、投递重试 |
| 实例从成功完整快照消失 | 标记 retired→停止 expected 和样本合成→结束实例告警 | 重新出现时清除 retired 并按新样本重新累计 |
| 性能健康查询 | 详情实时代理；列表读取进程内缓存并异步刷新 | 上游失败仅标记性能 unavailable，不影响本地用量和资源数据 |
| 快速任务历史 | probe/realtime/resource 完成后写 Redis 有界列表 | 写入失败不改变任务结果；Redis 不可用时 readyz 返回未就绪，查询返回内部错误 |
| 导出 | 后台任务使用同一统计服务生成 | 失败后按原条件重新创建 |

详细设计：[业务功能与平台 API](./多站点运营管理平台-详细设计-04-业务功能与平台API.md)。

## 9. 权限和安全基线

| 角色 | 权限 |
|---|---|
| admin | 查看、导出和全部管理操作 |
| viewer | 查看、筛选、下钻和导出，不允许远端用户搜索及任何写操作 |

安全基线：

- 密码至少 8 个 Unicode 字符且 UTF-8 编码不超过 72 字节，使用 bcrypt；
- 初始管理员密码由环境变量或首次启动一次性生成，不使用固定 `admin/admin`；
- 登录接口限流，并记录安全日志；
- 用户角色、状态、密码变化后立即使旧 Session 权限失效；
- Session Cookie 使用 HttpOnly、SameSite=Strict，生产环境启用 Secure；
- Access Token 和钉钉密钥使用 AES-256-GCM 加密；
- base_url 变更后不向新主机发送旧 Token，必须重新授权；
- HTTP 客户端禁止跨主机或降级协议重定向，并实施 SSRF 防护；
- 生产环境通过显式上游主机/CIDR 白名单、可信反向代理列表和可选企业 CA 文件约束网络边界；
- 日志、错误和 API 永不输出密码、完整 Token、Webhook 或签名密钥。

API 中数据库 BIGINT 主键、外键以及 quota、Token、请求数等计数统一返回十进制字符串；Unix 秒时间戳和分页页码保持 JSON number。前端不得把字符串 ID 转为 JavaScript `number`。

## 10. 调度和资源控制

平台为探活、实时状态、资源快照、小时用量、历史回填、校验、导出和通知使用独立队列。窗口型和手动任务必须具有持久化幂等键；周期 probe/realtime/resource 使用有界内存队列和单站互斥，执行结果另写 Redis 短期历史，不以 Redis 代替 `collection_run` 或恢复语义。

首期资源限制：

- 单用户最多 3 个 pending/running 导出任务；
- 全局最多 10 个 pending/running 导出任务；
- 单个导出文件默认最大 2 GiB，超过后失败并提示缩小范围；
- 手动补采单次最大范围默认 366 天，更大范围需拆分；
- 导出目录必须配置磁盘剩余空间阈值；
- 队列长度、最老任务等待时间、失败数和磁盘水位纳入监控。

具体限制允许通过系统配置调整，但不能配置为非正数或无上限。

## 11. 非功能目标

首期容量和验收基线：

| 指标 | 目标 |
|---|---|
| 站点规模 | 至少 50 个站点 |
| 纳管账户 | 至少 10,000 个 |
| 普通列表接口 | P95 小于 1 秒 |
| 31 天小时趋势 | P95 小于 3 秒 |
| Dashboard | 各区块独立加载，P95 小于 3 秒 |
| 上一完整小时采集 | 正常情况下 H+15 分前完成 |
| 调度恢复 | 进程重启后自动恢复未完成任务 |
| 数据正确性 | complete 窗口不得把未知数据表示为 0 |

性能验收使用固定容量画像：50 个站点、100,000 个远端用户、10,000 个纳管账户、30 天不少于 15,000,000 行 `usage_fact_hourly`、每站最多 100 个活跃通道和 200 个活跃模型；在 20 个并发只读用户下验证上述 P95。超出该画像需重新容量评审，不影响本期功能口径。

数据库备份、加密密钥备份、日志保留、磁盘告警和恢复演练属于部署必备项，即使不由业务开发模块直接实现，也必须在上线清单中明确责任人和执行方式。
