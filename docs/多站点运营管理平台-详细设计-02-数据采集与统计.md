## 7. 采集调度

### 7.1 周期任务

| 任务 | 周期 | 说明 |
|---|---|---|
| 在线探活 | 60 秒 | 更新在线状态、版本、汇率、数据导出开关 |
| RPM/TPM | 60 秒 | 仅保存当前值，不做历史快照 |
| 实例资源 | 60 秒 | 保存实例分钟级数据并评估资源告警 |
| 用户属性同步 | 每小时 | 按站点分页批量同步纳管账户属性 |
| 通道同步 | 每小时 | 同步通道 ID 和名称 |
| 小时用量 | 默认每小时 H+5 | 按 usage_delay_minutes 拉取上一个完整小时；生产认证画像不超过 5 分钟 |
| 次日校验 | 每日 02:00 | 重新校验前一天落在各站统计范围内的全部 expected 小时；普通完整日为 24 小时 |
| 周期校验 | 每周一次 | 重新校验最近 7 天 |
| 分钟数据清理 | 每日一次 | 清理超过留存期的分钟资源数据 |

### 7.2 默认并发

| 任务 | 并发上限 |
|---|---:|
| 在线探活 | 20 |
| RPM/TPM | 10 |
| 实例资源 | 10 |
| 用户/通道元数据同步 | 5 |
| 小时用量 | 5 |
| 历史回填 | 2 |

- 各站点随机错峰；
- 同一站点、同一任务类型使用进程内锁；
- 前一轮未完成时不重复启动；
- 手动补采进入同一任务队列；
- 并发数可由 admin 配置；
- 持久化任务使用 active_key 幂等约束，防止调度、手动操作和重启恢复重复入队。

### 7.3 重试策略

- 在线探活失败：等待下一周期，不立即重试；
- 资源和 RPM/TPM 失败：等待下一周期；
- 用户/通道同步失败：整轮不提交，等待下一小时；
- 小时用量失败：1 分钟、5 分钟、15 分钟重试；
- 历史回填失败：失败窗口重新入队，不阻塞其他站点；
- 401：立即标记授权过期，不继续重试；
- 429：遵循 Retry-After；
- 5xx 或网络错误：按对应任务规则重试；
- 站点离线后暂停资源、RPM/TPM 和小时采集，仅保留在线探活；
- 站点恢复后从游标补齐所有缺失小时。

## 8. 历史回填与数据完整性

### 8.1 首次站点回填

1. 使用 `/api/user/self` 完成授权身份校验；
2. 精确读取候选 root，校验 role=100、status=1、0<created_at<=当前时间；
3. 完整分页读取远端用户快照，校验分页一致性、候选 root 为最小正 ID 且 created_at 不晚于任一用户；读取失败、空快照或证明失败时接入失败；
4. 将已验证 Token、root_created_at 和 `statistics_start_at=floorToBeijingHour(root.created_at)` 原子保存，不提供人工确认或修改入口；
5. 检查 enable_data_export=true 及必要接口能力；
6. 从起点到上一个完整小时分段拉取全部可用历史；
7. 每个历史小时同时拉取 /api/data/flow 与 /api/data 并比较 request_count、quota、token_used；
8. 两个接口一致后按站点小时窗口写入基础事实，不一致时保留旧值或空窗口并标记 missing；
9. 生成站点、全局、模型、通道历史汇总；
10. 所有可恢复窗口 complete 后 statistics_status 变为 ready，存在 unavailable 或失败窗口时为 partial。

statistics_start_at 之前属于平台明确排除的统计范围，不进入完整率分母。起点首次写入后不可修改；重新授权必须重复完整分页、软删除、最小正 ID 与最早 created_at 的首用户证明，并核对 root_user_id/root_created_at 原值，不覆盖历史边界。发现不一致时标记 SITE_INCOMPATIBLE 并要求人工排查站点身份，不自动清理历史。

### 8.2 首次账户回填

1. 创建平台纳管账户；
2. 使用 site_id + remote_user_id 从基础事实中查找全部历史；
3. 生成账户 hourly/daily；
4. 重算所属客户 hourly/daily；
5. 若基础事实存在缺口，先执行站点补采；
6. 账户回填不按账户调用任何远端统计接口，只读取本地基础事实。

### 8.3 回填期间展示

- 新建操作立即返回；
- 对象状态显示 backfilling；
- 展示回填进度、已完成范围和最近错误；
- 首次站点历史回填和新账户首次回填按完成窗口逐步可见，已回填数据允许查询，API 和页面明确标记为部分数据；
- 账户 restore 与客户 enable 采用原子开放：恢复区间在全部本地重建成功前继续返回 paused/null，只展示暂停前历史、已完成范围和进度；全部成功后一次性清除暂停边界并开放整个恢复区间，失败不暴露半套归属统计；
- Dashboard 展示全局数据完整率；
- 回填完成后自动转为 ready；
- 失败自动重试，并允许 admin 手动重试；
- 不提供取消回填。

### 8.4 采集游标与任务记录

collection_cursor 保存：

- site_id；
- cursor_key，首期固定 usage；
- last_complete_hour；
- updated_at。

collection_cursor 只用于站点级连续小时事实，usage_hour 与 usage_backfill 共享同一 `cursor_key=usage`；usage_validation 只能回退该游标，并在缺口修复后按连续 complete 重新推进。账户和客户的本地历史重建进度记录在 collection_run，不使用独立游标。

周期性 probe/realtime/resource 的成功 tick 不逐次写 collection_run，只更新当前表、Prometheus 指标和结构化日志，并把最近执行结果写入 Redis 有界历史；失败导致状态变化时仍写一条 MySQL 诊断 run。手动刷新、全部窗口任务、本地重建和导出始终持久化。Redis 历史按站点与任务类型保留最近 N 条并设置 TTL，只用于运营排查，不参与恢复、告警状态机或业务统计；Redis 写入失败不得把已完成的快速任务改判失败。这样保留故障恢复所需证据，同时避免每年近亿条无业务价值成功 tick。

快速历史查询固定为 `GET /api/fast-tasks`，必须提供 `site_id` 和 `task_type=site_probe|realtime_stat|resource_snapshot`，可选 `status=success|failed`，使用 `offset/limit` 分页。返回 `items/offset/limit/total/has_more`，其中 `site_id` 为十进制字符串。Redis 不可用时接口返回统一内部错误，`readyz` 同时报告 `redis` 未就绪。

collection_run 保存：

- site_id；
- task_type；
- start_timestamp；
- end_timestamp；
- status；
- fetched_rows；
- written_rows；
- retry_count；
- priority；
- next_attempt_at；
- heartbeat_at；
- total_windows、completed_windows、failed_windows；
- created_request_id、last_request_id；
- error_code、error_params 和仅供技术详情的 error_message；
- started_at；
- finished_at。

一个 collection_run 表示一个逻辑任务范围，可以覆盖一个或多个小时；每个 run 对每小时的执行状态、尝试次数和下次重试由不可变归属的 collection_run_window 保存。collection_window 只表达事实当前是 pending/complete/missing/unavailable，不能兼任任务进度；因此 validation 可在不隐藏既有 complete 事实的情况下进入 pending/running。Worker 每次最多处理 24 个 run window 或运行 30 秒后主动让出并重新排队，使实时小时任务能够抢占长历史回填。页面回填进度和旧任务详情以 collection_run_window 为准，事实可见性以 collection_window 为准。

### 8.5 数据完整状态

| 状态 | 含义 |
|---|---|
| complete | 采集成功，数据完整 |
| partial | 聚合范围内仅部分站点或部分窗口完整，可展示已知合计但必须标记不完整 |
| pending | 窗口尚未完成或正在采集 |
| missing | 应采集但失败，数据未知 |
| unavailable | 上游无法提供该历史区间 |
| paused | 站点、客户或账户处于暂停统计状态 |
| backfilling | 正在历史回填 |

只有 complete 且基础事实无记录时才能补 0。missing、unavailable、pending 不得显示为零流量。跨站聚合存在部分完整数据时返回已知合计并标记 partial；若没有任何完整来源则数值返回 null。

统计接口返回：

- complete_site_count；
- expected_site_count；
- unit_type、complete_unit_count、expected_unit_count；
- completeness_rate；
- missing_site_ids；
- missing_ranges。

完整率公式：站点/全局/模型/通道单桶为 complete expected site 数 / expected site 数，跨多桶为 `complete_expected_site_hours / expected_site_hours`；账户为 `complete_expected_hours / expected_hours`，起点取 remote_created_at；客户为存在至少一个 expected 账户的 `complete_customer_site_hours / expected_customer_site_hours`。分母为 0 时 completeness_rate=1、data_status=complete，但数值是否为 0 仍按具体业务对象是否存在 expected 范围判断。paused、missing、pending、unavailable 均计入分母但不计入 complete 分子；statistics_end_at 之后不进分母。

summary 的 complete_site_count/expected_site_count 表示 as_of 所在最后一个时间桶的站点数，范围完整率必须使用 unit_count 字段，不能用 distinct site 数替代 site-hour 比例。

### 8.6 稀疏存储

- 无流量小时不写事实和汇总记录；
- 全天无流量不写 daily；
- 查询趋势时由后端补齐连续时间轴；
- complete 的缺失记录补 0；
- missing 或 unavailable 返回 null 及对应状态；
- 今日摘要必须精确查询今天，不能使用最近一条 daily 冒充今日。

## 9. 唯一事实源与窗口替换

### 9.1 基础小时事实粒度

usage_fact_hourly 唯一键：

~~~text
(site_id, remote_user_id, model_name, channel_id, use_group, token_id, node_name, hour_ts)
~~~

P0-A 上游契约固定参考 `new-api` commit `7c28993f6bd9e92616f3f578212577f8b7c40b45` 的 `model/usedata.go` 与 `model/usedata_flow.go`。`/api/data/flow` 的 `use_group`、`token_id`、`token_name`、`node_name` 与用户、模型、通道同属不可丢弃的原始统计身份。token_name 是 token_id 的展示快照，不参与唯一键；同一唯一键出现多个非空名称时使用确定性的字节序最小值，保证返回顺序变化不改变 canonical hash。

上游缺失可选字段时规范化为 use_group/node_name/token_name 空字符串、token_id=0，分别表示未知分组、未知节点和未知/已删除 Token。相同站点与小时只有八维唯一键完全相同的行才能求和；不得再跨 use_group/token_id/node_name 合并。

基础事实只保存远端身份，不保存 account_id 或 customer_id。

### 9.2 小时窗口整体替换

补采或重拉不能只 upsert 返回行，必须以站点小时为单位整体替换：

1. 拉取站点完整小时数据；
2. 按包含 group/token/node 的事实唯一键聚合并计算 `usage-fact-hourly-v2` canonical hash；
3. 开启 MySQL 事务；
4. 删除 site_id + hour_ts 的旧事实；
5. 插入本次完整结果；
6. 重算受影响的六级 hourly；
7. 重算受影响日期的基础 daily 和六级 daily；
8. 提交事务；
9. collection_run 标记 success。

成功返回空数组表示真实零流量，应删除旧窗口数据。请求失败时不得删除旧数据。

### 9.3 唯一事实源

usage_fact_hourly 是统计系统唯一事实源，但只有对应 collection_window.status=complete 的事实允许进入查询和汇总。校验失败时旧事实可以保留用于比较和恢复，但必须被完整性状态隔离，不能继续计入任何可见统计：

~~~text
usage_fact_hourly
    ├─ usage_fact_daily
    ├─ account_stat_hourly / daily
    ├─ customer_stat_hourly / daily
    ├─ site_stat_hourly / daily
    ├─ global_stat_hourly / daily
    ├─ model_stat_hourly / daily
    └─ channel_stat_hourly / daily
~~~

所有汇总采用重新聚合覆盖，不使用“旧值 + 增量”更新。任何汇总表都必须能够从 `usage_fact_hourly JOIN collection_window(status=complete)` 重建。窗口由 complete 退回 missing/unavailable 时，必须同步删除或重建该窗口汇总、重建受影响 daily/global，并把 collection_cursor 回退到缺口之前的最高连续 complete 小时。

## 10. 时间粒度与聚合规则

### 10.1 hourly

- 保存一个完整自然小时的增量；
- 有流量才写；
- 重拉后整体覆盖；
- hour_ts 为北京时间自然小时对应的 Unix 秒。
- statistics_start_at、statistics_end_at、手动补采边界和小时查询边界均必须对齐该整点；非整点请求返回 VALIDATION_ERROR，不做静默取整。

### 10.2 daily

- 当天每完成一个小时采集后立即重算；
- 表示截至最近完整小时的当日累计；
- 前端显示“截至 HH:00”；
- 次日 02:00 重新拉取并校验前一天全部数据；
- 校验完成后 is_final=true；
- 校验失败保持 is_final=false 并自动重试；
- 每周重新校验最近 7 天；
- 人工补采可重新打开并重算历史日期。

### 10.3 month 和 year

- 不单独建月表和年表；
- 请求数、quota、Token 从 daily SUM；
- 活跃账户从保留远端身份的 usage_fact_hourly/daily 重新去重；
- 月、年结果按查询实时生成。

### 10.4 页面查询限制

| 粒度 | 单次最大范围 |
|---|---:|
| 小时 | 31 天 |
| 日 | 2 年 |
| 月 | 20 年 |
| 年 | 全部历史 |

超过范围时提示切换到更粗粒度。Excel、CSV 导出不受页面范围限制。

所有统计 API 的 start_timestamp/end_timestamp 必须与 granularity 的北京时间桶边界对齐；hour 对齐整点，day 对齐 00:00，month 对齐月初，year 对齐 1 月 1 日。后端拒绝未对齐范围，前端日期选择器负责生成规范边界。

### 10.5 可加指标

- request_count；
- quota；
- token_used。

上述指标可以在同一统计口径内 SUM。

### 10.6 活跃账户去重

| 层级 | 计算方式 |
|---|---|
| 账户 | 有事实记录即活跃，不单独保存 active_accounts |
| 客户 | COUNT DISTINCT account_id |
| 站点 | COUNT DISTINCT remote_user_id |
| 模型 | 单站内 COUNT DISTINCT remote_user_id |
| 通道 | 单站内 COUNT DISTINCT remote_user_id |
| 全局 | COUNT DISTINCT site_id + remote_user_id |

账户统计仍对外返回 `active_users`，但不在 account_stat 表物化：查询范围内存在账户事实时返回 `1`，expected 范围全部 complete 且无事实时返回 `0`，expected 范围不完整时返回 `null`。这三个值来自事实和完整性联合派生，不能把“不保存字段”解释为“不适用”。

活跃数不能跨小时或跨天直接 SUM。日、月、年和任意区间都必须在相应范围重新去重。

# P0-B：上游日志明细采集边界

平台按站点采集官方 new-api `GET /api/log/` 管理员日志。固定参考基线为 `new-api` commit `7c28993f6bd9e92616f3f578212577f8b7c40b45`；该接口可用性依赖站点授权、管理员权限和上游 `log.consume_enabled` 开关。不可用、未授权或开关关闭时不伪造空日志，返回明确的 `unavailable`/`disabled` 完整性状态。

采集使用有限重叠时间窗口和 `p/page_size` 安全分页：每个窗口至少保留一个稳定重叠区间，按 `(created_at, upstream_log_id)` 去重；不得把 ClickHouse 展示 ID 当游标。请求带 request ID 和固定超时，窗口提交前后校验 site `config_version`，旧配置响应丢弃。日志不是财务事实，不参与 quota/request 汇总。

日志明细保留 `type/time/user/model/token/channel/group/request_id/upstream_request_id` 筛选维度。IP 默认不保存；content/other 仅保存脱敏后的受限摘要，移除 token、password、webhook、authorization、cookie、密钥和疑似凭据，禁止保存站点访问 Token。日志 TTL 由 `logs.retention_days` 配置控制，清理只删除日志事实，不影响 hourly/daily 财务事实和 collection run 审计。
# P0-C：全站用户库存

固定官方基线为 `new-api` commit `7c28993f6bd9e92616f3f578212577f8b7c40b45`，使用管理员 `GET /api/user/?p=&page_size=` 完整分页。平台只读取并保存 id、username、display_name、role、status、group、quota、used_quota、request_count、created_at、last_login_at 和 DeletedAt；email、password、access_token、OAuth/社交绑定、setting、邀请码和支付标识不得落库或进入日志/导出。

`user_sync` 一次成功快照同时更新全站库存和已纳管 account 交集，二者在同一 site/config_version 事务提交。分页失败、total/页大小漂移、重复 ID、超过 100000 用户、非法 bigint/字符串或 config fence 均整轮失败，不写部分库存、不增加 missing。完整轮中未出现的旧用户标记 missing，上游 DeletedAt 有效的用户标记 deleted；created_at 不同的相同 remote_user_id 标记 identity_mismatch，并与 account 暂停归属逻辑一致。

当前库存保留用户级余额、累计用量和最后登录；每小时固化按 site/role/status/group 的用户数、新增用户数、近期登录用户数、quota/used_quota/request_count，用于余额、增长和活跃趋势。库存统计不替代 usage_fact 财务事实。
# P0-D 渠道运营快照与统计契约

渠道采集固定对接官方仓库提交 `7c28993f6bd9e92616f3f578212577f8b7c40b45` 的管理员接口 `GET /api/channel/?p=&page_size=`。采集器必须完整分页，并只接收 `id/name/type/status/test_time/response_time/balance/balance_updated_time/models/group/used_quota/priority/weight/auto_ban/tag`。上游虽然可能返回 `key`、多 Key 状态、请求头覆盖、模型映射和其他私密配置，平台解码结构中不得声明这些字段，更不得读取、记录、持久化或导出。

一次成功 `channel_sync` 在同一配置版本栅栏事务中替换站点当前渠道快照并写入小时历史。当前表以 `(site_id, remote_channel_id)` 唯一；本次缺席的旧渠道标记 `missing` 并保留最后安全快照和历史，不物理删除。分页 total 漂移、重复 ID、任一非法字段、超限、超时或配置版本变化均使整个快照失败且不得部分提交。

余额使用规范化十进制定点字符串进入 DTO，数据库使用 `DECIMAL(30,10)`；不得经过二进制浮点。`used_quota`、渠道 ID 和所有计数使用 `int64`，对前端输出十进制字符串。小时历史支持余额、响应时间、可用渠道数/总渠道数与可用率趋势；只有完整快照才写 `complete` 历史。渠道余额和响应时间可作为告警稳定指标输入，但告警不得推断或展示任何 Key 数量、Key 健康或 Key 级状态。
# P0-E 性能指标历史与官方能力边界

官方基线固定为 new-api 提交 `7c28993f6bd9e92616f3f578212577f8b7c40b45`。标准接口 `GET /api/perf-metrics/summary?hours=` 用于枚举 model；`GET /api/perf-metrics?model=&group=&hours=` 返回 `model_name/series_schema/groups`，group 与 series 仅包含 `avg_ttft_ms/avg_latency_ms/success_rate/avg_tps` 和 bucket `ts`。官方内部数据库虽保存 request/success/latency_sum/ttft_sum+count/output_tokens/generation_ms counters，但标准 HTTP DTO 不暴露这些分子分母。

因此平台必须同时支持两类历史事实：`official_average` 原样保存官方单站 model+group bucket 平均值，保证单站能力不少于上游；`counter_ready` 预留真实 counters，只有未来标准端点明确返回 counters 时才启用。不得从 averages 反推请求数或分子。单站历史、筛选、site breakdown 和导出可使用 official_average；跨站总成功率、latency、TTFT、TPS 只有 counters 完整时才按分子分母重算，否则总值为 unavailable，并返回逐站原值 breakdown。

现有 `/api/sites/:id/performance` 继续实时代理 `/summary` 并使用 config_version 隔离的进程缓存；它不是历史事实，不参与历史聚合、导出或完整性判断。

# F1：充值订单与兑换码运营事实

基线固定为官方提交 `7c28993f6bd9e92616f3f578212577f8b7c40b45` 的管理员分页接口：`GET /api/user/topup` 与 `GET /api/redemption/`。两类数据都按完整分页快照采集，首尾 `total/max_id` 必须一致，重复 ID、空中间页、总数漂移、超过 100000 行或响应体上限均整轮失败且零部分提交。

充值仅保存远端 ID、用户 ID、`amount`、`money` 原始十进制文本、payment method/provider、创建/完成时间、状态和快照元数据。`money` 没有 currency，且不同 provider 的 amount/money 语义并不统一，因此禁止跨站金额合计或货币换算；仅提供订单数、状态、provider、site 统计，以及逐站/provider 的名义值。兑换码仅保存远端 ID、名称、状态、quota、创建/兑换/过期时间和使用用户 ID；`expired` 是 `status=enabled && expired_time!=0 && expired_time<now` 的 Clock 派生状态。

`trade_no` 与兑换码 `key` 属于禁止数据：不得声明于持久化模型、平台 DTO、日志、API、导出、fixture 或错误参数，不得通过 hash/encryption 变相保存。所有 ID/quota/amount 使用 int64，JSON 暴露为十进制字符串；money 使用严格 raw-number 解析后保存规范 decimal 文本。

# T1：上游异步业务任务

按 pinned Admin `GET /api/task/` 采集任务。平台仅保存 id/created_at/updated_at/task_id/platform/user_id/group/channel_id/quota/action/status/submit/start/finish/progress 与 `properties.model`；data、input、失败详情、结果 URL 和 private data 在解码边界丢弃。采集同时读取最近 48 小时重叠提交窗口并逐一重扫本地全部未终态 task_id，确保窗口外旧任务仍能收敛到 SUCCESS/FAILURE。统计以原始计数和毫秒/秒分子分母重算成功率、队列时间、运行时间和总耗时。
### D135 模型目录与渠道覆盖采集边界

每站点以管理员 `GET /api/models/` 全分页采集模型目录，仅保留 `id/model_name/description/icon/tags/vendor_id/status/sync_official/name_rule/created_time/updated_time`。分页必须校验最终 `total` 与最大 ID fence，最多 100000 行，并在站点配置版本锁内提交。`model_meta_sync` 为可恢复持久任务。渠道模型映射只从已冻结的完整 channel snapshot 的 `models/groups` 解析；只有完整快照允许原子替换映射，partial/unavailable 不删除旧边。缺失模型必须按名称精确连接，prefix/contains/suffix 规则不能冒充 exact。禁止采集 pricing、billing expression、自定义 endpoint 和 bound-channel enrich；icon URL 仅作为文本保存，不主动访问。
### D137 订阅计划目录边界

标准管理员端点为无分页 `GET /api/subscription/admin/plans`，响应元素包裹为 `{plan:{...}}`。平台只采 id/title/subtitle/price_amount/currency/duration_unit/duration_value/custom_seconds/enabled/sort_order/total_amount/quota_reset_period/quota_reset_custom_seconds/created_at/updated_at；price 使用 exact decimal，quota/seconds/ID 使用 bigint。显式丢弃 Stripe/Creem/Waffo product IDs、支付 payload、余额支付/溢出、购买限制和升级降级组等支付或策略配置。响应体受上限与 100000 行 hard cap，拒绝重复 ID；`plan_sync` 在配置 fence 内提交完整快照并维护 missing/reappear。标准 API 没有全局用户订阅或订单列表，因此本功能只是计划目录，禁止宣称订阅收入或用户订阅库存完整，也禁止逐计划 N+1 查询。
# D138 定价与分组目录采集及 parity 边界

每站点在同一 `pricing_group_sync` 配置栅栏任务中读取管理员 `GET /api/group/` 与 root 视角 `GET /api/pricing`。分组目录只保存规范化非空组名，因此能够保留尚无用量的已配置组；它与 usage fact 的 `use_group` 分组维度并列，禁止用“曾发生用量的组”冒充完整配置目录。定价目录冻结安全 DTO，仅保存模型名、vendor、模型价格 decimal 文本、group ratio decimal 文本、usable group 和 supported endpoint 标识；不得混入 D135 模型元数据目录，也不得保存渠道凭据、billing expression、请求头/参数覆盖、bound-channel enrich 或任意未冻结字段。

两个响应都必须满足响应体上限、类型校验、组名/模型键去重与站点 `config_version` fence；任一端点失败、重复、非法 decimal、未知结构或 fence 变化时对应资源整轮不提交、不增加 missing。完整轮以 `(site_id, remote identity)` 原子 upsert，并把旧目录中本轮未出现项标记 missing；重现恢复 normal。采集结果记录 `root` 可见性、`collected_at`、`config_version` 和逐资源 completeness，不得把 root 可见目录宣传为其他角色的权限视图。

固定基线下的权威不采集边界如下：`GET /api/token/` 只返回当前登录 root 自己的 Token，flow 中的 token 维度仅代表已观察到的使用身份，平台不宣称全站 Token 库存完整；`GET /api/performance/stats` 命中负载均衡后的随机进程并包含进程 heap/goroutine 与本地磁盘/缓存信息，站点资源以 instances 接口为权威；channel affinity 接口读取进程本地 cache，usage 端点还要求 key fingerprint，多实例下不能作为站点事实；Codex usage GET 代理未冻结外部 payload，401/403 时可能刷新并写回渠道 OAuth 凭据，禁止自动采集、代理或持久化；平台不执行远端渠道、Token、兑换码、订阅、任务等 mutation，只提供本文冻结的只读监控能力。

# D139 system-task 只读监控采集边界

固定官方基线为 `new-api` commit `7c28993f6bd9e92616f3f578212577f8b7c40b45`。每站点仅以 root 身份读取 `GET /api/system-task/list?limit=100` 和 `GET /api/system-task/current?type=`；不得读取单任务详情，不得调用 `POST /api/system-task/log-cleanup` 或任何远端任务提交、取消、重试、清理 mutation。`current` 只对 `log_cleanup`、`channel_test`、`model_update`、`midjourney_poll`、`async_task_poll` 五个白名单 type 各请求一次，用于补齐 list 未覆盖的 pending/running 任务，禁止对 list 每行逐任务查询形成 N+1。

状态白名单固定为 `pending/running/succeeded/failed`。解码边界只保留任务身份、type/status、created_at/updated_at、typed progress、typed result 和失败存在性；`active_key`、`locked_by`、payload/state/result 原始 JSON、error 原文以及所有未知字段必须在内存解码边界丢弃，不得进入日志。typed progress 只允许 nullable `total/processed/progress/remaining`；typed result 按 type 判别：log_cleanup=`deleted_count`，channel_test=`tested/succeeded/failed/disabled/enabled`，model_update=`checked_channels/changed_channels/detected_add_models/detected_remove_models/failed_channels/auto_added_models`，midjourney_poll=`unfinished_tasks/channels_scanned/null_tasks_failed`，async_task_poll=`unfinished_tasks/platforms_scanned/null_tasks_failed`。失败只落 `error_present` 和稳定本地 `error_code`：`UPSTREAM_SYSTEM_TASK_FAILED`、`UPSTREAM_SYSTEM_TASK_LEASE_EXPIRED` 或 `UPSTREAM_SYSTEM_TASK_INVALID_RESPONSE`，禁止复制或派生远端错误原文。

list 返回恰好 100 条时必须标记该轮 `partial`；返回 ID 序列存在 gap、重复、非严格倒序、非法 type/status/typed 字段或 body 超限时也不得宣称 complete。合法行仍可幂等保存，但本轮不得推进完整水位或把未出现的旧行判为缺失。current 的合法 active 行与 list 按 `(site_id, remote_id)` 合并；current 失败时本轮为 partial，不能把 list 中没有出现的 active 任务推断为终态。

`system_task_sync` 是持久、可恢复、按 site/config_version 栅栏的 metadata 任务。采集状态分别持久化 list 和五个 current resource 的 last attempt/success/failure、data_status、as_of、config_version 与安全本地 failure code；站点配置变化使旧响应以 `SITE_CONFIG_CHANGED` 丢弃。同步失败、进程重启和 lease 过期后从持久状态恢复，不依赖内存游标。terminal 行按平台 `system_task_terminal_retention_days` 和本地 `updated_at` 清理；pending/running 永不因年龄删除，直到后续 current/list 观察到终态。全局或站点 completeness 为参与站点/resource 状态的保守合成，任何截断、ID gap、current 失败、配置 fence 或同步失败均为 partial/unavailable，不能把已有事实隐藏为零。
