# 多站点运营管理平台 — 详细设计 01：总体架构与安全

> 上级文档：[多站点运营管理平台-概要设计.md](./多站点运营管理平台-概要设计.md)  
> 详细设计索引：[多站点运营管理平台-详细设计.md](./多站点运营管理平台-详细设计.md)

## 1. 背景与目标

公司运营多个独立部署的 new-api 站点。各站点具有独立服务器、数据库、用户、模型通道和运行实例，需要一个统一平台集中完成：

1. 站点录入、授权、在线状态和资源监控；
2. 指定账户的纳管及历史用量统计；
3. 客户与纳管账户的固定归属管理；
4. 账户、客户、站点、全局、模型、通道六个层级的统计；
5. 全局 Dashboard、详细统计、数据导出和告警通知；
6. 数据缺口检测、历史回填、次日校验和周期校验。

### 1.1 本期范围

- 账户、客户、站点和全局用量统计；
- 模型和通道维度统计；
- CPU、内存、磁盘和实例在线状态；
- 当前 RPM/TPM；
- Excel、CSV 导出；
- 钉钉机器人告警；
- admin、viewer 两级平台用户权限。
- 前端仅支持简体中文 `zh-CN`，不提供语言检测或切换，用户可见文案统一经 i18next 管理；
- MySQL 8.0+ 单实例部署，开发和验收使用 MySQL 8.4。

### 1.2 本期明确不做

- 财务对账、客户欠费、账单、付款、发票和调账；
- 在远端 new-api 创建用户、充值、调额、禁用用户，以及管理用户 API Token；
- 远端通道或财务数据写入；平台只提供已冻结只读数据的采集、统计、完整性和告警展示；
- 操作审计；
- 2FA、SSO 和复杂密码策略；
- 在产品页面中提供数据库备份、恢复功能；部署侧仍必须配置数据库和加密密钥备份；
- 多实例部署和分布式调度。

quota 及其换算金额仅用于运营展示，不作为财务结算依据。

## 2. 名词与统计口径

| 名词 | 定义 |
|---|---|
| 站点 Site | 一套独立部署的 new-api 服务 |
| 远端用户 Remote User | 某个站点中的 new-api user |
| 纳管账户 Account | 平台创建的统计对象，固定绑定一个站点远端用户和一个客户 |
| 客户 Customer | 平台中的业务客户，一个客户可拥有多个纳管账户 |
| 通道 Channel | new-api 中的上游通道，平台仅同步 ID 和名称用于统计展示 |
| 实例 Instance | new-api 的一个运行实例，通过 node_name 标识 |
| 采集 Access Token | 平台访问站点管理 API 使用的 root Access Token |
| 基础事实 | 从站点 /api/data/flow 按完整小时获取并按最小统计粒度归并后的数据 |
| 全局统计 | 所有站点全部用户的汇总统计 |

### 2.1 六个统计层级

| 层级 | 数据范围 |
|---|---|
| 账户统计 | 平台指定的纳管账户 |
| 客户统计 | 客户名下的纳管账户 |
| 站点统计 | 单个站点的全部用户 |
| 全局统计 | 所有站点的全部用户 |
| 模型统计 | 每个站点全部用户，按原始 model_name 汇总 |
| 通道统计 | 每个站点全部用户，按 channel_id 汇总 |

每个统计层级均保存 hourly 和 daily 两种物化数据。月、年粒度由 daily 动态聚合。

## 3. 部署架构

### 3.1 部署方式

- 独立后端服务：Go + Gin + GORM；
- 独立前端：React 19 + Base UI + Tailwind；
- 独立数据库：MySQL；
- 独立短期历史存储：Redis 7；仅保存快速任务有界执行历史；
- 单实例部署；
- 调度器与 API 服务运行在同一个后端进程；
- 生产构建将 Rsbuild 静态产物与 Go 二进制打入同一部署镜像，由 Go 服务同源提供 SPA 和 `/api`；开发环境继续由 Rsbuild 代理 `/api`；
- 不使用 Redis 分布式锁，不把 Redis 作为业务事实、持久任务或告警审计的权威存储；
- 平台故障不影响各 new-api 站点。

本平台不嵌入当前 new-api 服务。当前 new-api 源码只用于参考 Router → Controller → Service → Model 分层、上游 API 契约和 `web/default` 的视觉交互风格。平台使用独立代码库或独立工程目录、独立 Go Module、独立前端构建入口、独立数据库和独立发布流程，不复用当前 new-api 的 `/api/user/*` 路由或业务表。

静态服务规则：带内容 hash 的 `/assets/*` 使用一年 immutable cache；`index.html` 使用 no-cache；非 `/api` 且不存在的 GET/HEAD 路径回退 index.html 支持前端路由；`/api`、`/healthz`、`/readyz`、`/metrics` 永不走 SPA fallback。

~~~text
┌─────────────────────────────────────────────────────┐
│                多站点运营管理平台                    │
│                                                     │
│  React 前端 ── Go API ── 采集调度器 ── 告警引擎      │
│                    ├── MySQL                       │
│                    └── Redis（快速任务短期历史）   │
└───────────────┬───────────┬───────────┬────────────┘
                │ HTTP API  │           │
          ┌─────▼─────┐ ┌───▼──────┐ ┌──▼────────┐
          │ new-api 1 │ │ new-api 2│ │ new-api N │
          └───────────┘ └──────────┘ └───────────┘
~~~

### 3.2 数据库约定

- 仅支持 MySQL；
- 最低版本为 MySQL 8.0，开发、CI 和验收基线为 MySQL 8.4；
- 存储引擎使用 InnoDB；
- 字符集使用 utf8mb4；
- 时间统一使用 Unix 秒；
- 日期键使用北京时间 YYYYMMDD；
- quota、request_count、token_used 使用 bigint；
- 所有关键唯一键和查询索引显式定义；
- 核心业务和统计外键使用 RESTRICT，不使用级联删除。

## 4. 平台用户与权限

### 4.1 角色

| 角色 | 权限 |
|---|---|
| admin | 查看、导出及全部管理操作 |
| viewer | 查看所有模块和全部数据，允许导出，不允许写操作 |

敏感信息不向任何角色返回明文，包括密码哈希、站点 Access Token、钉钉密钥。

### 4.2 登录

- 仅支持用户名和密码登录；
- 密码使用 bcrypt 保存；
- 认证方式参考 new-api 的 Session 登录模式，但使用平台自己的用户表和路由；
- 不做 2FA、SSO、刷新令牌和复杂密码规则；
- 密码至少 8 个 Unicode 字符，UTF-8 编码不得超过 bcrypt 的 72 字节上限；
- 登录接口按来源 IP 和规范化用户名组合限流：10 分钟内最多 5 次失败，超过后阻断 15 分钟并返回 429 和 `Retry-After`；成功登录清除该组合的失败计数；计数保存在单实例进程内，重启后清空；
- 用户密码、角色或启用状态发生变化时递增 session_version，使旧 Session 立即失效。

### 4.3 初始管理员

首次启动通过环境变量 `PLATFORM_BOOTSTRAP_ADMIN_PASSWORD` 提供初始密码，或在标准输出中一次性生成随机密码：

~~~text
username: admin
password: <环境变量或一次性随机密码>
role: admin
must_change_password: true
~~~

初始化过程只保存 bcrypt 哈希，不在 SQL、日志或配置文件中保存明文。首次登录必须修改密码，之后才能访问其他页面。未提供环境变量且无法安全输出一次性密码时拒绝完成初始化。

### 4.4 平台用户功能

- 用户列表；
- 新增 admin 或 viewer；
- 修改用户名、显示名、角色；
- 启用、禁用用户；
- admin 重置用户密码；
- 用户修改自己的密码；
- 禁止禁用最后一个可用 admin；
- 不提供操作审计模块。

### 4.5 platform_user

| 字段 | 类型 | 说明 |
|---|---|---|
| id | bigint PK | 平台用户 ID |
| username | varchar(64) unique | 登录名 |
| password_hash | varchar(255) | bcrypt 哈希 |
| display_name | varchar(128) | 显示名 |
| role | varchar(16) | admin/viewer |
| status | tinyint | 1 启用，2 禁用 |
| must_change_password | tinyint | 是否必须修改初始或重置密码 |
| session_version | int | 密码、角色或状态变更时递增，用于使旧 Session 失效 |
| last_login_at | bigint | 最近登录时间 |
| created_at | bigint | 创建时间 |
| updated_at | bigint | 更新时间 |

初始化程序在事务内检查是否存在平台用户；仅在用户表为空时创建 bootstrap admin。重复启动不得重置密码，也不得覆盖已有管理员。

## 5. new-api 接入与授权

### 5.1 接入假设

- 所有站点由同一运营方控制；
- 所有站点由内部团队统一维护并保持接口契约兼容；生产部署不锁定版本号，接入和升级时验证 `/api/status` 及数据接口契约；
- 每个站点首个用户为 root，root 用户不可删除，其 created_at 早于或等于站点全部业务日志；
- 作为采集身份的首个且不可删除 root 用户不启用 2FA 或登录验证码；不得改用后建的 root/admin 账户；
- 站点必须开启 enable_data_export；
- enable_data_export 必须在首笔业务请求前开启，quota_data 不得做历史清理；该条件由站点部署清单保证，因为当前 API 无法反查历史开关时间；
- 平台不适配不兼容的旧版本。

### 5.2 授权流程

平台提供两种授权方式。

推荐方式为录入已有 Token：

1. admin 输入 root_user_id 和已有 root Access Token；
2. 平台调用 `GET /api/user/self` 校验返回用户 ID 与 root_user_id 一致、role=100、status=1；
3. 通过 `GET /api/user/:root_user_id` 精确读取候选 root，校验 created_at；
4. 完整分页读取包含已删除记录语义的远端用户列表，校验分页快照完整、root_user_id 为最小正 ID 且 root.created_at 不晚于任一用户；该首用户证明属于身份校验，失败时不保存 Token；
5. 在同一事务保存 root_user_id、加密后的 Access Token、root_created_at 和自动计算的 statistics_start_at；起点写入后不可修改；
6. 执行版本与接口能力检查，再同步通道并触发历史回填；能力检查失败时保留已验证授权和不可变起点，站点停在 pending_config/error 供修复后重试；步骤 4 的快照直接作为本次首次用户同步结果，不重复拉取。

可选方式为登录并重新生成 Token：

1. admin 输入 root 用户名和密码；
2. 页面明确提示 `GET /api/user/token` 在当前 new-api 实现中会生成新 Token 并覆盖原 Access Token；
3. admin 二次确认可能导致其他依赖原 Token 的系统失效；
4. 平台调用 POST /api/user/login，校验角色为 root；
5. 在同一临时登录会话中调用 GET /api/user/token；
6. 使用新 Token 调用 `GET /api/user/self` 完成身份校验，再精确读取候选 root 并通过完整用户快照证明其为首个用户；
7. 在同一事务保存 root_user_id、加密后的新 Access Token、root_created_at 和 statistics_start_at，随后立即释放密码和临时 Cookie Jar；
8. 执行能力检查、通道同步并进入回填流程；能力检查失败时按推荐方式的相同规则保留已验证授权和起点。

后续站点管理 API 使用：

~~~http
Authorization: <access_token>
New-Api-User: <root_user_id>
~~~

### 5.3 Access Token 安全

- Access Token 对称加密保存；
- 加密密钥由环境变量提供，不写入数据库；
- API 永不返回完整 Token；
- 日志和错误信息不得输出 Token；
- 重新授权时覆盖旧 Token；
- 解密失败，或必需的授权接口返回 401/403 时，auth_status 变为 expired；
- Access Token 与授权时的站点主机绑定；base_url 主机变化后不得向新主机发送旧 Token，必须重新授权；
- HTTP 客户端禁止跟随到其他主机、其他端口或更低安全协议的重定向；
- base_url 校验禁止 userinfo、fragment 和非 http/https 协议，并按部署网络策略限制可访问地址，防止 SSRF 和 DNS 重绑定。
- base_url 禁止 query；scheme 和 host 转小写、移除默认端口和尾斜杠后保存，路径保留大小写；连接前按系统设置 `upstream.allowed_host_suffixes`/`upstream.allowed_cidrs` 解析校验，连接时固定到已验证地址；两项为空时仅允许安全公网，私网必须显式命中 CIDR，特殊用途地址始终拒绝；TLS 证书校验不得关闭，企业 CA 仅通过 `UPSTREAM_CA_FILE` 注入。

网络白名单判定：配置了 host suffix 时主机名必须命中；配置了 CIDR 时每个实际连接 IP 必须命中。环回、链路本地、组播和云元数据地址默认拒绝；RFC1918 私网只有在 CIDR 中显式列出才允许。DNS 每次新建连接都重新解析并校验，HTTP 重定向重新执行完整校验。

### 5.4 本平台使用的 new-api 接口

| 用途 | 接口 | 周期 |
|---|---|---|
| 公开探活、版本、汇率、数据导出开关 | GET /api/status | 60 秒 |
| 校验当前授权身份 | GET /api/user/self | 授权和重新授权时 |
| 当前 RPM/TPM | GET /api/log/stat | 60 秒 |
| 实例资源 | GET /api/system-info/instances | 60 秒 |
| 全站小时明细 | GET /api/data/flow | 每个完整小时及历史回填 |
| 小时总量校验 | GET /api/data | 首次历史回填、次日及每周校验 |
| 远端用户列表 | GET /api/user/ | 授权首用户证明、每小时、首次回填 |
| 远端用户搜索 | GET /api/user/search | 添加纳管账户时 |
| 远端用户精确读取 | GET /api/user/:id | 创建账户提交前复核 |
| 通道列表 | GET /api/channel/ | 首次授权及每小时 |
| 性能健康滚动摘要 | GET /api/perf-metrics/summary?hours=N | 站点详情按需；列表 24 小时缓存缺失时异步刷新 |

本期仅使用用户、充值、兑换码和任务相关的已冻结只读接口；不提供远端用户或财务写入、兑换码密钥操作、用户 API Token 管理或远端任务 mutation。

性能摘要只允许 `1<=hours<=720`。详情请求实时代理单站上游；24 小时结果可写入按 `site_id + config_version` 隔离的进程内缓存，成功 TTL 1 小时、失败 TTL 5 分钟。该缓存不落 MySQL/Redis、进程重启即丢失，不构成历史事实或恢复承诺。

Redis 是生产和 Compose 的启动/就绪依赖。启动时必须能构造客户端；运行期 `PING` 失败时 `/healthz` 仍表示进程存活，但 `/readyz` 返回 503 且 `failed_checks` 包含 `redis`。快速任务本身不因历史写入失败而改写已经得到的业务结果；`GET /api/fast-tasks` 在 Redis 不可用时使用统一 `INTERNAL_ERROR` 响应。

## 6. 时间、金额与数值规范

### 6.1 时间

- 全平台统一使用 Asia/Shanghai；
- hourly 使用整点 Unix 秒；
- daily 按北京时间自然日计算；
- 查询区间统一使用左闭右开：[start_timestamp, end_timestamp)；
- 今日、本周、本月和年度均按北京时间计算。

### 6.2 quota 与金额

- 数据库只保存原始 quota；
- 不保存美元和人民币金额；
- 前端获取站点当前 quota_per_unit 和 usd_exchange_rate 后换算；
- 历史金额也使用站点当前汇率，因此站点配置变化后历史展示金额会变化；
- 金额仅用于运营展示，不用于财务结算；
- 跨站客户和全局数据必须保留 site_id 分项，前端逐站换算后相加；
- 导出时后端可使用站点当前汇率临时计算美元和人民币，但不落库；
- 导出文件注明“金额按导出时站点汇率计算”。

### 6.3 大整数精度

- API 中 quota、token_used、request_count 等 bigint 字段返回十进制字符串；
- 数据库 BIGINT 主键和外键也返回十进制字符串；Unix 秒时间戳、百分比、分页页码和分页总数保持 JSON number；
- 前端使用 Decimal 高精度计算；
- 表格和导出保留精确值；
- 图表展示时再转换为绘图数值。

### 6.4 非功能要求

| 类别 | 要求 |
|---|---|
| 独立性 | 平台故障不得影响任何 new-api 站点正常转发 |
| 只读优先 | 默认录入已有 Token，仅执行数据读取；可选的 Token 重新生成属于明确告知并确认的远端凭据变更 |
| 性能 | 列表和统计读取平台 MySQL，不在页面请求中实时穿透多个站点 |
| 调度隔离 | 单站失败不得阻塞其他站点，历史回填不得阻塞实时任务 |
| 数据一致性 | 小时窗口整体替换，所有汇总可从唯一事实源重建 |
| 完整性 | 缺口可识别、可补采、可校验，不把未知数据当成 0 |
| 安全 | Access Token 和钉钉密钥加密保存，密码使用 bcrypt |
| 可扩展性 | 新增站点仅需录入、授权和回填，不需要修改代码 |
| 可观测性 | 所有采集任务记录状态、窗口、行数、耗时和错误 |
| 前端体验 | 所有统计页面显示更新时间、数据状态和完整率 |

所有外部 HTTP 请求发送 `X-Request-ID`。来自页面的操作沿用入站 request_id；调度任务和重试为每次尝试生成新 request_id，并在 collection_run/投递日志中关联逻辑任务 ID。
