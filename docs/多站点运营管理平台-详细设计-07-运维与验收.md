## 45. 功能闭环

| 输入/事件 | 系统动作 | 可见结果 | 失败恢复 |
|---|---|---|---|
| 新建站点 | 保存→授权→校验 root→自动设置历史起点→能力检查→同步→全历史校验回填 | 部分数据→ready | 任务重试、手动补采、告警 |
| 站点断网 | 连续探活失败→offline；用量任务按有界重试失败并保留缺口 | 健康墙、告警和 missing/partial | 网络恢复后自动或手动按窗口补齐 |
| 站点换域名 | 修改 base_url→公开探活→暂停任务→重新授权→确认站点身份 | 原历史继续挂同一 site_id | 无法确认同一站点时新建 site |
| 小时无流量 | 空结果整体替换→window complete | 查询补 0 | 次日再次校验 |
| 小时采集失败 | 不删除旧事实→window missing | null/partial 和缺失范围 | 自动/手动补采 |
| 新建账户 | 绑定远端用户和客户→本地全历史重建 | 回填进度→完整账户统计 | 依赖站点缺口补采 |
| 账户归档/恢复 | 暂停账户/客户汇总→恢复区间重建 | paused→backfilling→active | 保留 paused_at 重试 |
| 客户停用/恢复 | 暂停客户和账户汇总→全区间重建 | 部分数据和进度 | 失败不清暂停边界 |
| 次日校验 | flow 与 data 双接口比对 | daily final | 不一致保留旧值并告警 |
| 汇率变化 | 更新 site 当前汇率 | 历史金额即时变化 | 原始 quota 不受影响 |
| 资源超阈值 | 连续评估→firing→钉钉 | 告警中心和通知 | 投递重试、恢复通知 |

| admin 初始登录 | 使用环境变量或一次性密码登录并强制改密 | 改密后进入 Dashboard | 中间件防绕过 |

## 46. 数据完整性闭环

数据完整性由四层共同保证：

1. collection_window 持久记录每站每小时是 pending、complete、missing 还是 unavailable；paused/backfilling 由站点生命周期和活动任务动态推导；
2. usage_fact_hourly 保存唯一事实，可重建全部汇总；
3. 小时整体替换避免旧行残留，source_hash 发现差异；
4. 首次历史回填、次日和每周任务均使用双接口校验持续修正历史。

任一汇总都能回答：

- 数据来自哪些站点；
- 预期有多少站点；
- 哪些窗口完整；
- 哪些窗口缺失或不可用；
- 当前数值是完整值、已知部分合计还是 null；
- 最近采集、重建和校验时间。

因此“稀疏存储”和“查询补 0”不会掩盖真实缺口。

## 48. 实现不允许自行改变的口径

1. 不得把 /api/data 当作基础事实接口；基础事实使用每小时 /api/data/flow。
2. 不得把 end_timestamp 当成上游排他结束；调用 new-api 时传 end-1。
3. 不得只 upsert 返回行；必须整站整小时替换。
4. 不得删除 collection_window；否则无法识别完整零流量。
5. 不得从任何汇总表反向修补事实。
6. 不得把 missing、unavailable 或 paused 转成 0。
7. 不得把 active_users 跨小时、天直接相加。
8. 不得把跨站 quota 用单一汇率换算。
9. 不得把账户/客户口径混入全站模型和通道物化表。
10. 不得在页面请求中实时串行访问多个 new-api 站点。
11. 不得保存 root 密码或返回站点 Access Token。
12. 不得为 viewer 暴露写接口。
13. 不得加入本期明确排除的财务、欠费、用户 Token、审计、2FA、SSO 或远端用户写操作。
14. 不得使用后建的 root/admin 账户替代站点首个且不可删除的 root；历史起点只能来自该首个 root 的 created_at。

| REDIS_DB | 非负整数数据库编号，默认 0 |

| FAST_TASK_HISTORY_RETENTION_SECONDS | 快速任务历史 TTL，正整数，默认 86400 |
| FAST_TASK_HISTORY_COUNT | 每站每任务类型最多保留条数，正整数，默认 100 |

| ENCRYPTION_KEY | Base64 编码的 32 字节 AES-256-GCM 主密钥 |
| PLATFORM_BOOTSTRAP_ADMIN_PASSWORD | 首次启动可选；至少 8 个 Unicode 字符且 UTF-8 不超过 72 字节，创建管理员后应从运行环境移除 |
| EXPORT_DIR | 独立、不可被 Web Server 直接静态访问的导出目录 |
| SESSION_COOKIE_SECURE | 生产环境必须为 true |
| PUBLIC_ORIGIN | 生产必填的 HTTPS origin，不含路径/query/fragment；用于 Origin 校验和通知链接 |
| TRUSTED_PROXIES | 可信反向代理 CIDR 列表；默认空，不信任任意转发头 |
| UPSTREAM_ALLOWED_HOST_SUFFIXES | 允许接入的 new-api 主机后缀列表，可空但不能与 CIDR 同时为空 |
| UPSTREAM_ALLOWED_CIDRS | 允许连接的解析地址 CIDR 列表，可空但不能与主机后缀同时为空；显式配置后才允许私网地址 |
| UPSTREAM_CA_FILE | 可选企业 CA PEM 文件；不得关闭 TLS 证书校验 |
| DINGTALK_ALLOWED_HOSTS | 可选附加钉钉 Webhook 域名，默认仅 oapi.dingtalk.com |
| METRICS_ALLOWED_CIDRS | production 必填的 `/metrics` 来源 CIDR 列表 |
| TZ | 固定 Asia/Shanghai |

缺失数据库、Redis、Session 密钥、加密主密钥、导出目录不可写或上游网络白名单为空时，服务拒绝进入 ready 状态。production 还必须校验 PUBLIC_ORIGIN 为 HTTPS、SESSION_COOKIE_SECURE=true、TRUSTED_PROXIES 和 METRICS_ALLOWED_CIDRS 语法合法。密钥和密码不得通过命令行参数传入。

部署主机和 MySQL 必须启用时间同步，平台主机与标准时间偏差不得超过 5 秒；超过时由基础设施告警并暂停发布，因为小时边界、Token 签名和任务超时均依赖可靠时钟。

`OLD_ENCRYPTION_KEY`、`NEW_ENCRYPTION_KEY` 仅允许在停机执行 `secrets reencrypt` 时临时注入，不属于常规服务环境变量，完成后必须移除。正常运行、`verify-backup` 和 `verify-restore` 只读取 `ENCRYPTION_KEY`；三个变量都必须是 Base64 编码的 32 字节密钥。

当前二进制提供以下运维命令契约；这是实现说明，不代表对应 planned 验收已经执行：

| 命令 | 当前行为 |
|---|---|
| `new-api-pilot migrate` | 只加载运行配置、连接 MySQL 并执行带 checksum/checkpoint 的 migration；不启动 HTTP、seed 或 Scheduler，成功退出 0，失败退出非 0 |
| `new-api-pilot secrets reencrypt [--dry-run] [--batch-size=100]` | batch-size 范围 1～1000；使用 OLD/NEW 维护密钥检查或分批暂存后，以单个最终事务和源 hash CAS 原子切换密文，支持同一密钥对断点续跑 |
| `new-api-pilot verify-backup --mode=manifest-only --manifest=/absolute/path/manifest.json` | 不连接数据库，完整校验 manifest 未知/缺失字段、路径、双 sidecar、dump hash/size、SOURCE 坐标及 `ENCRYPTION_KEY` 指纹，供导入前 fail-closed preflight 使用 |
| `new-api-pilot verify-restore --mode=full --manifest=/absolute/path/manifest.json` | 只读校验 manifest/sidecar、密钥指纹、migration、表/seed/外键、全部密文、关键计数、窗口/游标/active_key 和六级聚合不变量 |

`secrets reencrypt`、`verify-backup` 和 `verify-restore` 的 stdout 固定为单个版本化 JSON 报告，不输出密钥或明文秘密；退出码 0=全部检查成功、1=运行或一致性检查失败、2=参数/维护配置错误。运维自动化必须同时保存 JSON 和退出码，不能只解析人类日志。

### 50.2 健康与监控

后端提供：

- `GET /healthz`：进程存活，不检查远端站点；
- `GET /readyz`：MySQL、Redis、migration、Scheduler 初始化完成且加密密钥有效；任一检查失败返回 503、`status=not_ready` 和稳定排序的 `failed_checks`，Redis 失败项固定为 `redis`；
- `GET /metrics`：Prometheus 文本格式，仅允许来源 IP 命中 METRICS_ALLOWED_CIDRS，不包含站点 URL、用户名、Token 或 Webhook 标签；
- 内部指标：HTTP/上游延迟和错误率、数据库连接池、各队列长度、最老任务等待时间、任务结果、窗口完整性、告警状态和投递、导出任务/耗时/字节、运行组件就绪与恢复、导出目录容量；
- 日志统一包含 request_id、site_id、task_type、target 和 range，不包含敏感凭据。

上述 label 只能取代码中的固定白名单值，未知值统一归为 `other`；禁止用站点 ID、用户 ID、模型/通道名、URL、request ID、错误文本或其 hash 扩展 label。数据库采样失败不得中断业务 runtime，也不得用失败轮次的零值覆盖上一份成功快照；导出文件系统不可读时必须写 `total=0` 和 `free=0`，由容量异常规则显式告警。

备份任务通过 `PROMETHEUS_TEXTFILE_DIR` 配置的 node-exporter textfile 目录原子暴露 `new_api_pilot_backup_last_success_timestamp_seconds`、`new_api_pilot_backup_last_failure_timestamp_seconds`、`new_api_pilot_backup_failures_total`、`new_api_pilot_backup_last_duration_seconds` 和 `new_api_pilot_backup_last_size_bytes`。应用指标采样器每分钟读取 MySQL `UNIX_TIMESTAMP()`，暴露 MySQL 时钟减应用时钟的 `new_api_pilot_clock_offset_seconds`。基础设施告警使用独立于本平台钉钉 Worker 的 Alertmanager receiver，避免“钉钉投递故障仍只通过同一投递链路告警”。

必须配置的运维告警基线：

- 生产认证配置下上一完整小时在 H+15 分仍未完成；非认证配置按 `hour_end + max(15,delay+10)` 监控且已被发布门禁阻断；
- 任一任务队列最老等待时间超过 15 分钟；
- 导出目录剩余空间低于 5 GiB 或 10%；
- MySQL 连接失败或连接池持续耗尽；
- Redis 不可达或 readyz 持续包含 redis；
- 钉钉连续投递失败；
- 调度器停止创建周期任务。

最低 PromQL 使用完整指标名：

- `time() - new_api_pilot_scheduler_heartbeat_timestamp_seconds > 180` for 2m；
- `new_api_pilot_task_oldest_age_seconds > 900` for 5m；
- `new_api_pilot_collection_lag_seconds > 900` for 5m；
- `(new_api_pilot_export_free_bytes < 5 * 1024^3) or (new_api_pilot_export_free_bytes / new_api_pilot_export_total_bytes < 0.10)` for 5m；
- `sum(new_api_pilot_db_connections{state="in_use"}) / sum(new_api_pilot_db_connections{state="max"}) > 0.90` for 5m；
- `increase(new_api_pilot_alert_deliveries_total{channel="dingtalk",result="failed"}[10m]) >= 3` for 5m；
- `time() - new_api_pilot_backup_last_success_timestamp_seconds > 3600` for 5m，或 `increase(new_api_pilot_backup_failures_total[1h]) > 0`；
- `up{job="new-api-pilot"} == 0` 或 `absent(up{job="new-api-pilot"})` for 2m；`absent(new_api_pilot_scheduler_heartbeat_timestamp_seconds)` for 2m；
- `absent(new_api_pilot_backup_last_success_timestamp_seconds)`、`absent(new_api_pilot_clock_offset_seconds)` for 5m；这些监控链路缺失告警必须走前述独立基础设施 receiver；
- `abs(new_api_pilot_clock_offset_seconds) > 5` 立即，用于检测 MySQL 与应用宿主时钟偏移；`new_api_pilot_ready == 0 or absent(new_api_pilot_ready)` for 2m。

部署仓库在 [`deploy/prometheus`](../deploy/prometheus/README.md) 保存可加载的 recording rules、告警规则和抓取示例，并在 [`deploy/grafana`](../deploy/grafana/README.md) 保存运维 Dashboard；`make check-prometheus` 使用固定 Prometheus 镜像执行 `promtool check rules`。发布证据必须另行记录 Prometheus 规则加载成功 API 输出和 Dashboard 导入结果，仓库文件或 CI 通过本身不等于受控环境验收完成。若文件系统 total=0/不可读，容量异常规则必须 firing，不能除零静默通过。

### 50.3 备份和恢复

- MySQL 每日全量备份，并按部署要求增加 binlog 或更高频增量备份；
- 生产基线 RPO 不超过 1 小时、RTO 不超过 4 小时；全量备份保留 30 天，binlog 至少覆盖最近 7 天；
- `ENCRYPTION_KEY` 必须进入受控密钥备份，数据库备份与密钥备份分开保管；
- 备份 manifest 的 `encryption_key_id` 按 §28.6 取原始 32 字节密钥 SHA-256 小写十六进制；密钥保管系统以该指纹映射不可变 secret version。任何仍被保留的全量/binlog manifest 所引用的旧版本都不得销毁，保留期至少覆盖最长备份保留期 30 天并延续到最后一份引用备份到期后的下一次成功恢复演练；
- SESSION_SECRET 可轮换，轮换后允许全部平台 Session 失效；
- 加密主密钥轮换必须停服务、先备份数据库和旧密钥，临时注入 `OLD_ENCRYPTION_KEY`/`NEW_ENCRYPTION_KEY`，先 dry-run 再运行 `new-api-pilot secrets reencrypt`；只有 JSON status=success、退出码 0、暂存清空且全部密文可由新密钥读取后，才把运行时 `ENCRYPTION_KEY` 切到新版本；禁止直接替换环境变量；
- 每季度至少执行一次恢复演练，验证站点 Token、统计事实、任务状态和告警配置可恢复；
- 导出临时文件无需备份，可从 export_job 条件重新生成。
- 应用结构化日志默认保留 30 天；安全相关日志（登录限流、密码/角色/站点授权/敏感配置变化）保留 180 天。日志保留不等同于产品内操作审计模块。

### 50.4 容量与保留

首期容量验收固定使用：50 个站点、100,000 个远端用户、10,000 个纳管账户、30 天不少于 15,000,000 行 `usage_fact_hourly`、每站最多 100 个活跃通道和 200 个活跃模型。以 20 个并发只读用户执行普通列表、31 天小时趋势和 Dashboard P95；小时采集验收假设单站上一小时接口 P95 不超过 60 秒，并验证 H+15 前完成。超出画像后必须重新评估索引、分区和并发。

P95 测试在与生产同规格的 MySQL 8.4 和单实例应用上执行：预热 2 分钟、持续采样至少 10 分钟，每类接口不少于 1,000 次成功请求，统计服务端耗时且不计客户端网络；错误响应不从样本中静默剔除，错误率必须低于 0.1%。

| 数据 | 保留 |
|---|---|
| 业务 hourly/daily | 永久，达到容量阈值后评审分区或归档，不得无计划删除 |
| collection_window | 永久 |
| collection_run | 窗口/手动/失败/本地重建永久；成功自动 metadata run 保留 30 天；90 天后可清理永久 run 的详细错误文本 |
| 资源 minutely | 默认 90 天 |
| 资源 hourly/daily | 永久 |
| 告警事件 | 永久 |
| 导出文件 | 默认 24 小时 |

“永久”保留存在一个明确例外：从未形成业务事实/统计且满足 §29.8 物理删除条件的对象，其无历史价值 collection 元数据和 resolved 告警可在删除事务中一并清理；只要形成过事实、统计、firing 告警或成功/失败的业务范围任务，就不再允许物理删除该站点。

### 50.5 首次部署、升级与回滚 Runbook

发布物是带不可变版本号和 digest 的单一镜像；生产 compose/编排文件、反向代理配置和环境变量清单进入受控部署仓库，秘密使用文件/Secret 注入，不提交 Git。单实例升级允许短暂停机，不声称滚动无损。

当前生产构建由 Go 二进制嵌入前端资源。SPA fallback 仅处理 GET/HEAD；`/api`（包含未注册 API）、`/healthz`、`/readyz`、`/metrics` 及其子路径始终保留给后端，其他方法不 fallback。`index.html` 和未带内容哈希的资源使用 `Cache-Control: no-cache`，带至少 8 位内容哈希的资源使用 `public,max-age=31536000,immutable`，并提供 ETag/条件请求；反斜杠、NUL、`.`/`..`、隐藏路径段和非法 bundle 一律拒绝，不能回退到 index 暴露内容。

首次部署：

1. 运维确认 MySQL 8.4、Asia/Shanghai、时间同步、独立数据库和账号；创建仅限该 schema 的 DML+DDL 权限账号，准备权限 0700 的 EXPORT_DIR 持久卷。
2. 配置 §50.1 全部生产变量，确认站点接口契约和导出能力均已验证；执行 `docker compose config` 并人工检查输出中没有明文秘密进入归档。
3. 拉取并校验镜像 digest；执行 `docker compose run --rm api /usr/local/bin/new-api-pilot migrate`。该命令只运行版本/校验和/checkpoint 检查与 migration，成功为 0，失败不得启动 API。
4. 执行 `docker compose up -d api`，依次验证 `/healthz`=200、`/readyz`=200、`/metrics` 仅允许监控网段、登录后 `/api/user/self` 正常。
5. 反向代理只暴露 HTTPS 业务入口，转发 Host/Proto，关闭任意 CORS；验证 HTTP 跳 HTTPS、Secure/HttpOnly/SameSite Cookie、`index.html` no-cache、hash 资产 immutable、`/api` 不走 SPA fallback。
6. 创建测试 viewer，完成登录/改密/403、站点无 Token 预检和一次导出冒烟；移除 bootstrap 密码后归档版本、digest、migration、时间和执行人证据。

停机升级：

1. 宣布维护窗口，停止新任务并等待最多 30 秒，`docker compose stop api`。
2. 按 §50.6 创建带一致性 binlog 坐标和校验值的升级前全量备份；应用已停止后记录“迁移边界”的 binlog file/position，同时备份当前镜像 digest、schema_migration 版本、部署配置和对应 `ENCRYPTION_KEY` 非敏感版本 ID。
3. 使用新镜像执行一次性 `migrate`，成功后启动新 api；验证 ready、schema_migration、seed 幂等、登录、站点授权解密、采集队列和统计查询。
4. 观察至少一个 60 秒状态周期和一个受控任务；任一必需检查失败立即进入回滚，不带故障继续开放流量。

回滚：migration 采用前向版本，不提供临时手写 down SQL。只有发布清单明确标记“旧应用兼容新 schema”时才允许直接切回旧镜像；默认流程为停止新应用、恢复升级前全量备份，并从该备份坐标之后只回放到已记录的迁移边界坐标之前，注入旧版本密钥引用、启动旧镜像并重复首次部署的健康/登录/采集核验。禁止用模糊墙钟时间越过迁移边界。回滚失败时保持服务停止并升级事故，不允许在未知 schema 上反复启动不同版本。

### 50.6 备份、PITR 与恢复 Runbook

备份客户端通过权限 0600 或 0400 的 `/run/secrets/mysql-client.cnf` 取凭据，禁止把密码写入命令行和日志；凭据只允许来自环境、defaults file 或标准输入。仓库提供 `scripts/backup.sh`，不接收命令行参数，要求 `BACKUP_ROOT` 为绝对无符号链接路径，并从环境读取 `MYSQL_DATABASE`、`IMAGE_DIGEST`、`ENCRYPTION_KEY`，每日全量示例：

~~~bash
BACKUP_ROOT=/srv/backups/new-api-pilot \
MYSQL_DATABASE=new_api_pilot \
MYSQL_DEFAULTS_FILE=/run/secrets/mysql-client.cnf \
IMAGE_DIGEST='sha256:<64-hex>' \
ENCRYPTION_KEY="${ENCRYPTION_KEY}" \
bash scripts/backup.sh
~~~

脚本先通过持续 MySQL 连接取得与 migration runner 相同的 `GET_LOCK('new-api-pilot:migration-runner', 60)`，并在 `--single-transaction --source-data=2` dump、server identity 和完整 schema_migration/checksum 采集全部完成后才释放，禁止并发 migration 让快照与 manifest 分叉。脚本解析与一致性快照对应的 `CHANGE REPLICATION SOURCE TO SOURCE_LOG_FILE=..., SOURCE_LOG_POS=...` 注释；把 file/position、UTC 时间、镜像 digest、完整 schema_migration 版本与 checksum、MySQL 版本/server UUID、dump 大小/SHA-256 和按 §28.6 算出的完整 encryption_key_id 写入 `manifest.json`，并为 dump 与 manifest 分别生成受限 basename 的 sidecar。任一坐标、字段、密钥长度或 checksum 缺失即失败；成功目录先在同一文件系统暂存，最后原子 rename 发布。JSON stdout 只输出 12 位密钥指纹。使用 GTID/托管快照时必须在 manifest 记录等价的 executed GTID set，不得混用两套起点。

MySQL 开启 ROW binlog；备份侧使用 `mysqlbinlog --read-from-remote-server --raw --stop-never` 或等价托管能力持续归档，上传延迟不得超过 15 分钟。全量、binlog 和 SHA-256 manifest 使用备份系统加密后存入与应用主机隔离的对象存储；每日自动校验 checksum，每周在隔离库试恢复，失败通过独立基础设施告警通道通知。

PITR 恢复：

1. 记录事故发现时间、目标恢复时点、开始时间和执行人；停止 api，禁止写入原库。
2. 创建空 MySQL 8.4 恢复实例，校验时区、utf8mb4/utf8mb4_unicode_ci 和 READ-COMMITTED；目标 schema 必须存在且零表。导入前执行 `new-api-pilot verify-backup --mode=manifest-only --manifest=/absolute/path/manifest.json`，复用生产 validator 一次校验绝对路径、未知字段、全部 manifest 字段、密钥指纹、dump 与双 sidecar 的 hash/size 及 SOURCE 坐标；失败时目标必须仍为零表。只恢复全量且无需回放 binlog 时，可直接使用 `scripts/restore.sh` 一次完成本步和第 4 步；该脚本读取绝对 `RESTORE_MANIFEST`、绝对 `RESTORE_RELEASE_ROOT`、`MYSQL_DATABASE`、`DATABASE_DSN`、`ENCRYPTION_KEY` 和 defaults file。
3. 从 manifest 的 SOURCE_LOG_FILE/SOURCE_LOG_POS 之后开始按文件顺序回放；首文件使用 `--start-position`，目标时点恢复显式执行 `TZ=UTC mysqlbinlog --stop-datetime='YYYY-MM-DD HH:MM:SS' ... | mysql ...`，其余文件不得重复起点事务。升级回滚不用 stop-datetime，而以迁移边界 file/position 精确停止。检查 schema_migration 无缺口、核心表行数非负、外键检查通过。
4. 完成全部 binlog 回放后，按 manifest encryption_key_id 从受控密钥系统注入精确匹配的 `ENCRYPTION_KEY`，执行只读 `new-api-pilot verify-restore --mode=full --manifest=/absolute/path/manifest.json`：校验 manifest/sidecar 与 dump、密钥指纹、全部密文可解、migration/checksum/progress、表/seed/外键、关键计数、collection_window/run-window、游标、active_key 和六级 hourly/daily 聚合不变量。退出码或 JSON 任一失败都不创建 release gate；全部通过后才原子发布含 `verify-report.json` 的 release 目录。`restore.sh` 在全量恢复路径中强制执行相同校验并原子 rename gate；PITR 不得在 binlog 回放完成前调用该一体化脚本发布。编排只允许在 gate 存在后启动备份对应镜像，再验证 ready、admin 登录、至少一个站点授权任务、pending/running 父子任务接管、告警和配置可读。生产恢复不得依赖测试 fixture 总量。
5. 切换流量后记录最后一条已恢复业务时间与恢复完成时间。`数据损失窗口<=1 小时` 且 `开始到业务核验完成<=4 小时` 才判定 RPO/RTO 通过；否则演练失败并整改。

任一步 checksum、binlog 顺序、密钥匹配或核心核验失败都必须停止，不覆盖唯一备份，不把未核验实例提升为生产。导出文件不恢复，成功任务文件缺失后按 EXPORT_FILE_MISSING 重新生成。

验收工具链是进入开发后的首批基础设施任务，不冒充当前已完成实现：实现阶段必须增加 `make docs-check` 校验 Markdown 链接、D/R/A 连续性及逐 D 映射、MessageRef catalog/params、单一 `zh-CN` 资源和 manifest 完整性；增加 `cd web && bun run test:e2e` 执行 Playwright 页面验收；增加 `make acceptance` 串联 docs-check、Go 测试、`bun run check`、E2E 和受控演练证据检查。在这些命令实际存在并通过前，文档可以判定“开发就绪”，但项目不得判定“实现完成”或通过 §52 发布门禁。当前文档评审使用与 `docs-check` 同断言的只读机械检查并记录结果，首个功能 PR 前必须把它固化为上述命令。
