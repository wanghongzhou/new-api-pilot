## 28. 与 new-api 现有架构的对齐

本平台独立部署，以下内容只说明参考的 new-api 架构模式，不表示把平台代码直接写入当前 new-api 服务：

- router/api-router.go：按资源建立 Gin RouterGroup，并在路由层组合认证中间件；
- middleware/auth.go：Session 身份、角色和请求上下文统一在中间件处理；
- controller/user.go：登录后保存 id、username、role、status 等 Session 字段；
- common/gin.go、common/page_info.go：仅参考响应与分页形态；平台在自身 `common/response.go` 和 `common/page.go` 实现 success/message/code/data/request_id/field_errors 与 PageData；
- common/json.go：业务代码通过 common.Marshal、common.Unmarshal、common.DecodeJson 操作 JSON；
- model/main.go：GORM 初始化、连接池、迁移和种子初始化；
- controller/usedata.go、model/usedata.go、model/usedata_flow.go：小时 quota_data 和流量导出字段；
- controller/channel.go、model/channel.go：通道分页、ID、名称和状态；
- controller/system_info.go、service/system_instance.go：实例状态和资源字段；
- web/default：参考共享 Axios、TanStack Router、TanStack Query、DataTablePage、SectionPageLayout 和 VChart 的组织方式与视觉风格。

平台拥有独立 Go Module、main 入口、配置、数据库和前端工程。仅复用架构思想和必要的通用实现方式，不导入 new-api 的业务包，不注册到当前 new-api Router，也不复制支付、Token、OAuth、2FA、订阅和中继代码。

### 28.2 平台 Session 认证

认证沿用 new-api 的简单 Cookie Session 模式：

1. POST /api/user/login 先执行来源 IP + 用户名组合限流，再校验用户名和 bcrypt 密码；
2. Session 写入 id、username、role、status、must_change_password、session_version；
3. Session 内容使用 SESSION_SECRET 做 HMAC-SHA256 完整性保护，验签失败一律按未登录处理；Session 不包含密码、Token 或其他秘密；
4. Cookie 名称为 session，Path=/，Domain 留空形成 host-only cookie，MaxAge=2592000，HttpOnly=true，SameSite=Strict；
5. Secure 由 SESSION_COOKIE_SECURE 配置，生产 HTTPS 必须为 true；
6. 前端 Axios withCredentials=true，并从本地用户状态附加 New-Api-User；
7. Auth 中间件同时校验 Session id 与 New-Api-User 一致，每次请求读取 platform_user 当前 role、status、must_change_password 和 session_version；
8. 平台 API 不支持 Access Token、JWT 或远端站点 Token 作为平台登录凭证；
9. POST /api/user/logout 清空 Session；
10. 用户被禁用后，下一次请求立即返回 401；
11. 修改或重置密码、修改角色、启用或禁用用户时递增 session_version，使旧 Session 全部失效；当前操作者修改自己密码时重新建立当前 Session。

登录限流按 `client_ip + ':' + normalized_username` 建立内存桶。用户名 trim 后转小写，10 分钟内第 6 次失败开始阻断 15 分钟；阻断响应为 HTTP 429、code=LOGIN_RATE_LIMITED，并返回 `Retry-After`。成功登录清除该组合计数。Gin 只信任 `TRUSTED_PROXIES` 中反向代理提供的转发头，默认不信任任何代理。

平台用户名保存为小写 ASCII，允许 3～64 位字母、数字、点、下划线和连字符；display_name trim 后为 1～128 个 Unicode 字符。最后一个启用 admin 的降级/禁用检查必须在事务中锁定全部启用 admin 行，避免并发请求同时通过检查。

Session 存储 platform_user.id，不依赖用户名，因此修改用户名不会使 Session 身份错配。

### 28.3 强制修改初始密码

must_change_password=true 时，ForcePasswordChange 中间件仅允许：

- GET /api/user/self；
- PUT /api/user/password；
- POST /api/user/logout。

其他 API 返回 HTTP 403、code=PASSWORD_CHANGE_REQUIRED。前端 _authenticated 根路由检测该字段并跳转 /change-password；直接输入其他地址也不能绕过后端限制。

### 28.4 角色中间件

| 中间件 | 允许 |
|---|---|
| UserAuth | admin、viewer |
| AdminAuth | admin |

viewer 的导出接口使用 UserAuth。所有 POST、PUT、PATCH、DELETE 管理接口使用 AdminAuth；无需为两个角色引入通用 RBAC 引擎。

### 28.5 JSON、事务和数值约束

- 所有 JSON 编解码通过 common 包装函数；
- GORM 是默认数据库访问方式，聚合 SQL 必须使用参数绑定；
- quota、Token、请求数在 Go 中使用 int64；解析上游 int 时先转 int64 并拒绝负数；
- API bigint 使用字符串 DTO，禁止直接把 int64 JSON 数字发送给前端；
- MySQL 事务隔离级别使用 READ COMMITTED，窗口替换时显式锁定 collection_window；
- 所有外部 HTTP 请求都携带 request_id，并在日志中记录 site_id、task_type、range，不记录密码、Access Token、Webhook 或签名密钥。
- 入站 `X-Request-ID` 仅接受 1～64 位 ASCII 字母、数字、点、下划线和连字符；缺失或非法时生成新的随机 ID，响应头和 ApiResponse.request_id 使用同一值。

### 28.6 密钥和敏感配置

- SESSION_SECRET 必须由环境变量提供，解码后至少 32 字节；缺失时拒绝启动；
- APP_ENV=production 时 PUBLIC_ORIGIN 必须是无路径、无 query/fragment 的 HTTPS origin，用于写请求 Origin 精确校验和通知链接；
- TRUSTED_PROXIES 可选显式列出可信反向代理 IP/CIDR；为空时 Gin 不信任任意 X-Forwarded-*，production 直连部署不得被强制配置代理；
- 上游允许域名后缀、允许 CIDR、连接/响应/请求/导出超时以及每 origin 限流/并发均存入 platform_setting 并由系统设置热更新，不再由环境变量提供；两个上游白名单均为空时仅允许普通公网地址，私网地址必须显式命中允许 CIDR；loopback、link-local、multicast、云元数据与其他特殊用途地址始终禁止，不能通过设置解除；UPSTREAM_CA_FILE 仅作为可选只读文件挂载加载企业 CA，禁止 InsecureSkipVerify；
- 内部维护的上游站点不要求版本号锁定；站点 `/api/status` 必须满足状态响应契约，`version` 字段允许为空（自维护 fork 可不发布版本号），其余状态字段仍必须存在且类型正确，数据导出能力必须显式开启；
- `ENCRYPTION_KEY` 必须是 Base64 编码的 32 字节密钥；
- 非敏感 `encryption_key_id` 固定为解码后 32 字节密钥的 SHA-256 小写十六进制（64 字符）；日志、备份 manifest 和轮换证据只记录该指纹，不记录 Base64 密钥。密钥保管系统必须以该指纹唯一映射到不可变 secret version；
- 站点 Access Token、钉钉 Webhook 和 secret 使用 AES-256-GCM；
- 每次写入生成 12 字节随机 nonce，存储格式 v1:base64(nonce):base64(ciphertext)；
- AAD 分别使用 site:{site_id}:access_token 和 setting:{setting_key}，防止密文跨记录替换；
- 发布物提供仅停机维护时使用的 `new-api-pilot secrets reencrypt [--dry-run] [--batch-size=100]` 子命令：从 `OLD_ENCRYPTION_KEY` 解密、用 `NEW_ENCRYPTION_KEY` 重加密全部站点 Token 和非空敏感 setting，两个变量都必须是 Base64 编码的 32 字节密钥且指纹不同；常规服务启动不读取这两个维护变量；
- 非 dry-run 按短事务分批写入 0005 暂存表，暂存项只保存 AAD 身份、源密文 SHA-256、新密文和是否需更新，不保存明文；全部暂存并用新密钥自检后，在一个最终事务中按源 hash CAS 更新所有业务行，任一行变化则整体回滚，同事务清空暂存并把 job 标为 complete；相同 old/new 指纹可断点续跑，不同组合拒绝接管 active job；
- `secrets reencrypt` 的版本化 JSON 只输出 old/new 指纹前 12 位、匿名计数和稳定错误码，不输出完整指纹、密钥、明文或密文；
- 解密失败不得回退明文，站点 auth_status=expired 或配置显示 decrypt_error；
- bcrypt cost 固定为 10；密码至少 8 个 Unicode 字符且 UTF-8 编码不超过 72 字节，前后端使用相同字节长度规则，避免 bcrypt 截断或拒绝；
- 平台只允许同源前端，Cookie SameSite=Strict，CORS 默认关闭；production 下所有 POST/PUT/PATCH/DELETE `/api` 请求（含登录）必须携带与 PUBLIC_ORIGIN 精确一致的 Origin，缺失或不一致返回 403 ORIGIN_FORBIDDEN。healthz/readyz/metrics 不属于浏览器写接口。

上游主机后缀先执行小写、去末尾点和 IDNA ASCII 规范化；仅 `host == suffix` 或 `host` 以 `.` + suffix 作为标签边界结尾时命中。禁止直接使用字符串 HasSuffix，以免 `notexample.com` 命中 `example.com`。IP 字面量不参与 suffix 匹配，只能命中显式 CIDR。

同源 Web 响应统一设置 `X-Content-Type-Options: nosniff`、`Referrer-Policy: no-referrer`、`X-Frame-Options: DENY` 和只允许本源脚本/样式/连接的 Content-Security-Policy（允许项随实际静态构建清单测试，不使用 `unsafe-eval`）；生产反向代理设置 HSTS。文件下载使用 `Content-Type: application/octet-stream` 与 attachment，不内联用户可控内容。

## 29. 完整 MySQL 结构

### 29.1 版本与迁移原则

- MySQL 最低版本 8.0，开发、CI 和验收固定使用 MySQL 8.4；检测到更低版本时拒绝启动；
- 默认字符集 utf8mb4、排序规则 utf8mb4_unicode_ci、引擎 InnoDB；
- 项目上线前只维护 `0001_initial_schema.sql` 完整基线，所有表、列、索引、外键和检查约束直接落入该文件，不保留开发期增量迁移；上线后如需变更才从 `0002_xxx.sql` 开始追加前向迁移。GORM AutoMigrate 仅允许测试环境使用，不作为生产迁移机制；
- migration runner 使用 MySQL `GET_LOCK('new-api-pilot:migration-runner', 60)` 串行执行，并使用 schema_migration 的 version/checksum 防止漏执行或被修改；同一版本 checksum 变化时拒绝启动；
- migration SQL 一旦进入冻结版本或发布镜像即不可修改；任何后续 DDL/DML 变化必须新增更高版本 migration。没有可核验的历史 checksum 和等价结构证明时不得添加兼容 allowlist，未知 checksum 一律拒绝；
- schema_migration_progress 按版本保存 checksum、下一条 statement_index 和 ready/dirty 状态。每条 DDL 执行前先持久化 dirty checkpoint，执行后验证该版本、该语句注册的结构 postcondition，再推进 checkpoint；进程在 DDL 提交与 checkpoint 更新之间退出时，重启会先验证 postcondition，成立则继续，否则重放该幂等 DDL；
- MySQL DDL 会隐式提交，因此不得宣称整个 migration 天然原子。非 DDL 语句按单条语句连同 checkpoint 放在同一事务中；全部语句完成后才原子写 schema_migration 并删除 progress。当前基线的每条建表 DDL 都按语句顺序绑定表存在性 postcondition，缺失时拒绝执行；
- 默认配置和告警规则由启动后的幂等 Seed Service 初始化，不依赖外部 init.sql；bootstrap admin 由启动程序根据环境变量或一次性随机密码创建；
- 枚举值由 constant 包和 Service 校验，不依赖 MySQL CHECK；
- 以下 DDL 是字段、索引、外键和默认值的唯一依据。

alert_event.active_key 在 pending/firing 时写为 rule_key + ':' + target_type + ':' + canonical_target_key；resolved 时置 NULL。canonical_target_key 固定为：site=`site_id`，instance=`site_id/node_name`，account=`account_id`；collection 按 §34.6 的具体 rule_key 使用 `site_id/hour_ts` 或 `site_id/run_id`，不使用含糊的通用格式。同一 rule_key 的 Warning/Critical 以及 global/site override 因而只能存在一个活跃事件，事件上的 rule_id 记录本次实际生效规则。上述 nullable unique key 利用 MySQL 8.0 允许多个 NULL 的唯一索引语义，保证同一任务或事件只能有一个活跃记录，同时允许保留任意数量的终态历史记录。

### 29.8 删除约束检查

#### 29.8.1 D141 维护与实例生命周期迁移

初始 schema 创建 `data_maintenance_state`，冻结五个 operation 的持久状态、北京时间日期水位、复合游标、`scope_revision CHAR(64) CHARACTER SET ascii COLLATE ascii_bin NOT NULL DEFAULT ''`、安全错误码、授权 site/config scope 与 run 引用。operation-specific CHECK 要求：`authorize_pricing_group_sync` 必须 site/config 非空、`date_key=0` 且使用 versioned site scope，scope_revision 为空；两个 retention operation 的 scope_revision 为空；两个 resource operation 在 `date_key>0` 的 pending/running/complete/failed 状态均必须保存 64 位小写 hex revision。其余四项必须 site/config 为空、`scope_key='global'`，并限制 request/run 字段只出现在授权 intent。迁移 postcondition必须核对 22 列、3 个索引、2 个 RESTRICT/SET NULL 外键和全部实际 CHECK；`run_id` 的正值由指向 `collection_run(id)` 的外键保证，不能同时增加与 `ON DELETE SET NULL` 冲突的 CHECK。

初始 schema 直接创建实例生命周期区间表，不从 `site_instance` 合成历史区间。schema contract 必须核对 evidence/generated open 列、`uk_site_instance_lifecycle_start`、`uk_site_instance_lifecycle_open`、range 索引、site RESTRICT 外键和 range/evidence CHECK；partial pre-created schema、dirty checkpoint、重复迁移和 checksum 变化都必须确定性失败或恢复。`legacy_unknown` 仅保留给明确标记的外部历史导入或恢复数据，不得据此声称连续在线。site 删除时 lifecycle 属于可显式删除的 owned metadata；`data_maintenance_state` 仅 pending/running 活动 intent 阻断，complete 及可安全终止 failed 状态在同一删除事务显式清理，FK RESTRICT 仅作最后防线。

Service 在执行物理删除前先检查：

| 对象 | 必须全部为 0 |
|---|---|
| site | account、usage_fact_hourly/daily、全部业务统计和资源历史行 |
| customer | account、customer_stat_hourly/daily |
| account | account_stat_hourly/daily |

所有对象还必须不存在 pending/running collection_run 和 firing alert_event。site 满足删除条件时，Service 在同一事务内显式删除其 site_channel、单站 alert_rule override、尚未产生事实的 collection 元数据、当前 site_instance 和未形成历史价值的已恢复告警事件，再删除 site；customer/account 在无统计且无活动任务/告警时同样显式清理无历史价值的关联元数据。不使用 ON DELETE CASCADE。数据库外键作为最后防线，Service 返回 DELETE_RESTRICTED 并列出阻止删除的关联类型。平台用户本期只启用/禁用，不提供物理删除。
