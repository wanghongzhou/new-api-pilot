## 30. new-api 上游接口契约

### 30.1 NewApiClient

每个站点对应一个逻辑客户端，底层共享带连接池的 http.Client：

- ConnectTimeout 5 秒；
- ResponseHeaderTimeout 15 秒；
- 普通读取总超时 30 秒；
- 小时数据导出总超时 120 秒；
- MaxIdleConns 100、MaxIdleConnsPerHost 10；
- 自动去除 base_url 尾斜杠；
- 管理 API 固定发送 Authorization: 原始 access_token 和 New-Api-User: root_user_id；
- 不加 Bearer 前缀；
- Access Token 与授权时的 scheme、host、port 绑定；base_url 主机信息变化后客户端拒绝发送旧 Token；
- 禁止跟随到其他主机、其他端口或由 HTTPS 降级到 HTTP 的重定向；
- URL 不允许 userinfo、query 和 fragment；scheme/host 小写、默认端口和尾斜杠移除后保存，路径保留大小写；DNS 解析结果按部署 allowlist 校验，并在连接时防止 DNS 重绑定；
- NewApiClient 每次调用只执行一次 HTTP 尝试，不做任何隐式重试；幂等读取的网络/429/5xx 重试全部由 collection_run/run-window Worker 统一计数和退避。`GET /api/user/token` 虽为 GET 但会修改远端 Token，Worker 和客户端均绝不自动重试；
- 401 分类为 auth_expired，403 分类为 permission_denied，429 分类为 rate_limited，5xx 分类为 upstream_error；
- 所有响应先校验 HTTP，再校验 success 字段；
- 所有请求发送 `X-Request-ID`；页面触发任务沿用入站 ID，调度和每次重试生成新的尝试 ID；
- 响应体最大 64 MiB，超过直接失败并记录 UPSTREAM_RESPONSE_TOO_LARGE。

#### 30.1.1 new-api-latest 限流基线与平台请求预算

平台以 `new-api-latest` 当前源码默认值作为兼容基线；部署方可以提高上游限额，但平台默认配置不得假设上游已经放宽。上游限流均在业务认证前后之外独立生效，命中后返回 HTTP 429；其中通用 IP 限流的响应没有可靠 `Retry-After`，平台必须自行进入保守冷却，不能立即并发重放。

| 上游限制 | 默认值 | 计数键与覆盖范围 | 对平台的影响 |
|---|---:|---|---|
| Global API | 360 次 / 180 秒 | `GA + ClientIP`；覆盖 `/api/**` 及 legacy dashboard API | 平台所有探活、授权、分页快照、用量、资源和目录请求共享同一预算，是采集控制的硬上限 |
| Global Web | 120 次 / 180 秒 | `GW + ClientIP`；Web 路由 | 平台不调用，不计入采集预算 |
| Critical | 20 次 / 1200 秒 | `CT + ClientIP`；登录、敏感 mutation、`/api/usage/**` 和 `/api/log/token` 等显式挂载路由 | 当前平台只读管理接口不经过该层；新增接口前必须核对路由中间件 |
| Search | 10 次 / 60 秒 | 已认证 user ID；Token/日志搜索 | 当前平台不调用 |
| Email verification | 2 次 / 30 秒 | ClientIP | 当前平台不调用 |
| Upload / Download | 各 10 次 / 60 秒 | ClientIP；当前源码只保留 limiter 工厂，未挂载到平台使用的路由 | 当前平台不调用；后续若启用远端文件接口须重新评审 |
| Model request | 默认关闭；开启后全局默认成功 1000 次/分钟、总次数 0 表示不限，并可按用户组覆盖 | 已认证 user ID；仅 relay 模型请求路由 | 平台不调用 relay，不计入管理 API 预算 |
| HTTP 连接池 | MaxIdleConns 500、MaxIdleConnsPerHost 100 | 上游 relay 客户端连接复用，不是请求次数限流 | 不能据此提高平台请求并发；管理 API 仍受 Global API 限流 |
| SQL 连接池 | MaxIdleConns 100、MaxOpenConns 1000 | new-api 进程数据库连接池 | 不是平台可消费的并发承诺，平台仍需限制在途管理请求 |

平台为每个规范化上游 origin 维护统一请求 governor，所有 `NewApiClient` 实例和所有站点共享，不允许按任务、站点或 client 实例各自计算预算。系统设置默认 `upstream.rate_limit_requests=300`、`upstream.rate_limit_window_seconds=180`，即最多每 600ms 放行一个请求，保留 60 次/180 秒余量给浏览器、人工运维和未知调用；默认 `upstream.max_inflight_per_origin=4`，匀速门控与在途门控必须同时满足。预算参数允许管理员按已核验的上游配置热更新，但必须为正整数，且窗口换算后的放行间隔至少 10ms；默认不得高于上述兼容基线。

governor 采用 `routine:bulk = 3:1` 加权公平队列：授权、探活、实时、资源、元数据和实时小时用量为 routine，历史回填与历史验证为 bulk；一侧为空时另一侧可使用全部槽位。每次请求在真正写出 HTTP headers 前取得许可，直到响应体完成校验后才释放在途名额。429 带合法 `Retry-After` 时冻结该 origin 至指定时间且最长一小时；不带或非法时至少冻结完整的配置窗口。冻结期内新任务只排队，不消耗 HTTP 尝试次数，也不允许同一批失败窗口同时再次打到上游。队列等待受调用方 context 约束，取消后必须移除等待者且不得泄漏在途名额。

### 30.2 授权 DTO 与 Cookie Jar

推荐的已有 Token 授权：

~~~text
request: {
  mode: "existing_token"
  root_user_id: int64
  access_token: string
}
~~~

平台使用受保护的只读接口验证 root 权限和数据导出能力，不调用 `/api/user/token`，因此不会修改远端凭据。

可选的登录并重新生成 Token：

~~~text
POST /api/user/login
request:  { username: string, password: string }
response: { success, message, data: { id, username, display_name, role, status, group } }

GET /api/user/token
headers: Cookie: 登录会话；New-Api-User: data.id
response: { success, message, data: access_token }
~~~

已有 Token 模式和生成 Token 模式最终都必须使用管理 Header 调用 `GET /api/user/self`，确认响应 id 等于 root_user_id、role=100、status=1。授权过程使用临时 Cookie Jar 保持登录和取 Token 的会话连续性；流程结束后不再持有密码和 Cookie Jar。当前 new-api 的 `GenerateAccessToken` 会生成随机 key、调用 `SetAccessToken` 并更新 user 记录，因此 GET /api/user/token 会覆盖远端 root 的管理 Access Token。该模式必须由 admin 明确选择并二次确认，且不得在普通探活中调用。

### 30.3 GET /api/status 映射

| 上游字段 | 平台字段 |
|---|---|
| version | site.version |
| system_name | site.system_name |
| quota_per_unit | site.quota_per_unit |
| usd_exchange_rate | site.usd_exchange_rate |
| enable_data_export | site.data_export_enabled |

TCP/TLS/HTTP 可达且返回 2xx 时判定本次网络探活成功；连续 3 次网络或 HTTP 失败后判定 offline，任一次成功立即恢复 online。若 2xx 响应的 JSON/envelope/必需字段不兼容，则站点仍为 online，但能力检查记为 incompatible、statistics_status=error。成功取得两个正汇率时更新 last_rate_at；探活失败或本次汇率字段非法时保留最近一次有效值。授权状态不由公开 status 接口推断。

授权和每次探活都校验接口响应契约及数据导出能力。内部维护的站点不做版本号锁定；契约校验失败时能力记为 incompatible、停止需要授权的数据采集并触发告警。

### 30.4 小时事实接口

P0-A pinned upstream baseline 为 `new-api` commit `7c28993f6bd9e92616f3f578212577f8b7c40b45`：`/api/data/flow` 每行除 user_id/username/model_name/channel_id/count/quota/token_used 外，还映射 `use_group`、`token_id`、`token_name`、`node_name`。use_group 最大 128、token_name 最大 255、node_name 最大 128 个 Unicode 字符；token_id 为非负 int64。字段缺失按空字符串/0 规范化，非法 UTF-8、越界或负 token_id 使整轮失败。canonical、窗口替换和事实 hash 必须包含 use_group/token_id/node_name，并把 token_name 作为确定性展示快照写入 hash，禁止旧 v1 hash 与新 v2 数据混用。

基础事实必须调用 root 可见的 GET /api/data/flow，不能使用 GET /api/data 代替。原因是 /api/data 仅按 model_name + created_at 聚合，缺少 remote_user_id 和 channel_id，无法形成已确认的事实唯一键。

上游 FlowQuotaData：

| 字段 | 使用方式 |
|---|---|
| user_id | remote_user_id |
| username | username_snapshot |
| model_name | 原样保存 model_name；空字符串也原样保存，前端显示“未提供模型名” |
| channel_id | channel_id，缺失或 0 保留为 0 |
| count | request_count |
| quota | quota |
| token_used | token_used |
| token_id | 忽略并跨 token 求和 |
| use_group | 忽略并跨 group 求和 |
| node_name | 忽略并跨实例求和 |
| channel_name | 仅可作为临时显示，不作为事实字段 |

当前 new-api 的 flow 接口不返回 created_at，而是对查询范围聚合。因此平台每次只查询一个完整小时：

~~~text
平台窗口：[H, H + 3600)
上游参数：start_timestamp=H
          end_timestamp=H + 3599
~~~

new-api 查询条件的结束时间是包含关系，禁止直接传 H+3600，否则会把下一个小时的整点数据混入当前窗口。

每行校验：user_id>0、channel_id>=0、count/quota/token_used>=0；username 和 model_name 必须是合法 UTF-8 且不超过 255 个 Unicode 字符，model_name 允许空字符串。任何数值溢出、非法 UTF-8 或越界字符串使整个小时失败，不截断、不部分写入；重复维度行在内存中用 int64 安全加法聚合，溢出则失败。

### 30.5 总量校验接口

首次历史回填、次日和每周校验额外调用 GET /api/data：

- 按单小时请求；
- 取 model_name、created_at、count、quota、token_used；
- 将 /api/data/flow 结果按 model_name 汇总；
- 比较 request_count、quota、token_used；
- `/api/data` 是按模型小时总量的权威来源，`/api/data/flow` 是可归属到用户、渠道、分组、Token 和节点的明细来源。平台先严格校验两套响应，再按模型分别计算 `data - flow` 的 request_count、quota、token_used 残差；三个指标均不得为负，且 checked int64 运算不得溢出。
- 某模型存在非负残差时，平台为该模型补一条 `remote_user_id=0`、`username_snapshot=历史未归属`、`use_group=__legacy_unattributed__`、channel/token/node 为零值的历史未归属事实。该保留用户 ID 不允许来自真实 flow，残差事实参与站点、模型、渠道和全局用量总量，但不计入 active_users，也不进入账户或客户汇总。
- 补齐后必须再次逐模型及整小时校验三项指标与 `/api/data` 完全守恒；flow 中存在 data 不存在的非零模型、flow 任一指标大于 data、非法字段、负数或运算溢出仍使窗口标记 missing，保留旧事实，不执行替换，并创建 DATA_VALIDATION_MISMATCH 告警。flow/data 均为空时视为该小时确实无数据并成功完成窗口。
- collection_window.attribution_status 持久记录 `attributed` 或 `legacy_unattributed`。验证任务按同一规则重算 source_hash 和归属质量；哈希与质量均未变化时只更新 verified_at，否则原子替换事实并重建汇总。

该双接口校验还能发现上游 flow 接口因历史数据 use_group 为空而漏行的情况。首次授权能力检查从最近 24 个完整小时按新到旧寻找第一个有流量窗口执行同样比对；没有流量时能力检查记为 skipped，待首次采集到非空完整小时后立即补做。skipped 不阻止确认统计起点，但页面必须显示“首次有流量后补验”。

### 30.6 远端用户 DTO

GET /api/user/?p=N&page_size=100 和 GET /api/user/search?keyword=... 返回 data.page、data.page_size、data.total、data.items。平台仅映射：

- id；
- username；
- display_name；
- role；
- status；
- group；
- quota；
- used_quota；
- request_count；
- created_at；
- last_login_at；
- `DeletedAt.Valid`（当前受支持上游 DTO 的原字段名），仅用于区分软删除用户和证明首用户；不向平台前端透传上游删除结构。

password、access_token、email、OAuth、Token 列表等字段全部忽略。单轮同步记录第一页 total，最多读取 ceil(total/100)+2 页；只有所有页成功、ID 无冲突且最终累计结果满足完整快照校验时才能提交。items 提前为空、total 异常变化或超过保护页数时整轮失败，不据此把旧用户标记为 missing。

创建纳管账户提交时，平台必须再调用 `GET /api/user/:id` 精确读取所选用户并核对返回 ID；搜索结果只用于选择，不作为最终写入快照。搜索结果为软删除用户时不得展示为可选，精确读取不存在、已删除或返回 ID 不一致时返回 UPSTREAM_USER_NOT_FOUND/CONFLICT；用户名、分组、role、status、额度等发生变化时使用最新值创建，非启用状态允许纳管但会按规则告警。

本项目业务不变量规定首个用户是不可删除的 root 用户。平台在授权时同时使用精确读取和一轮完整用户快照：候选 root 必须 role=100、status=1、created_at 位于 (0,now]、ID 为包含软删除记录的快照中的最小正 ID，且 created_at 不晚于快照内任一用户。任一页失败、total 漂移、ID 冲突、空快照或证明失败均返回 SITE_INCOMPATIBLE 且不保存 Token。重新授权重复同一证明，并核对 root_user_id/root_created_at 与已存值完全一致。

### 30.7 通道 DTO

GET /api/channel/?p=N&page_size=100 映射 id、name、status。平台同步全部状态通道；远端本轮不存在的旧通道只标记 remote_missing=1，不删除名称，以保证历史统计可读。通道接口不会返回 key，平台也不得请求 key 接口。

### 30.8 实例 DTO

GET /api/system-info/instances 返回：

- node_name；
- status；
- stale_after_seconds；
- started_at；
- last_seen_at；
- info.node、info.role、info.runtime、info.host；
- info.resources.cpu.usage_percent；
- info.resources.memory.usage_percent；
- info.resources.storage.total_bytes、used_bytes、free_bytes、used_percent。

上游实例 `status` 的首期合法值只有字符串 `online`、`stale`；缺失、非字符串或其他值使整轮返回 UPSTREAM_RESPONSE_INVALID 且不写任何成功样本，禁止把新值直接透传。数据库/API 的 upstream_status=`unknown` 仅用于从未取得合法实例状态的本地占位/迁移记录，不代表可接受的上游枚举。平台保存合法上游 status 和 stale_after_seconds 作为原始诊断字段；页面 effective_status、site.health_status 与告警统一使用当前生效的 `instance_stale` 规则阈值判断，默认 90 秒、单站规则可覆盖。每 60 秒采样一次，同一分钟重复拉取时以最新结果覆盖。

实例快照校验：node_name 必填、1～128 且同一响应内唯一；百分比必须是有限数并位于 0～100，字节数不得为负。重复 node_name 或结构无法解析使整轮失败且不写样本；单个可选资源指标缺失时该指标写 null，不据此触发阈值告警。

### 30.9 RPM/TPM DTO

GET /api/log/stat 返回 data.quota、data.rpm、data.tpm。平台只保存 rpm、tpm 和 last_realtime_stat_at；quota 不写入业务统计。超过 120 秒未成功更新时前端标记当前值过期。

性能健康使用 `GET /api/perf-metrics/summary?hours=N`，其中 N 为 1～720。响应映射每个模型的 model_name、request_count、success_rate、avg_latency_ms、avg_tps；request_count 必须为非负 int64，比例和均值必须为有限非负数，同一响应内 model_name 不得重复。任一模型字段非法使整次响应失败，不返回部分列表。平台按 request_count 加权计算总成功率、平均延迟和平均 TPS，不得直接平均各模型百分比；详情结果是上游滚动摘要，不进入用量事实、资源表或历史聚合。

### 30.10 上游字段边界

所有上游字符串先验证 UTF-8 和字符数，所有 JSON number 先以可检测溢出的方式解析到 int64/decimal，不能等到数据库报错后再分类：

| DTO | 必需边界 |
|---|---|
| status | version 1～64、system_name 0～128；quota_per_unit/usd_exchange_rate 必须为有限正 decimal(30,10)；enable_data_export 必须为 boolean |
| user | id>0；username 1～255、display_name 0～255、group 0～128；role/status 为 int32；quota/used_quota/request_count 为非负 int64；created_at 位于 (0,now+5s]；DeletedAt 必须为 null 或含合法 Valid boolean 的结构 |
| channel | id>0；name 0～255；status 为 int32；同一完整快照 ID 不得冲突 |
| instance | node_name 1～128；hostname 0～255、runtime_version 0～64、goos/goarch 0～32；started_at/last_seen_at 非负且不超过 now+5s；百分比 0～100；字节数非负 int64 且 used<=total；stale_after_seconds 1～3600 |
| log/stat | rpm/tpm/quota 为非负 int64；业务只使用 rpm/tpm |
| perf-metrics/summary | hours 1～720；model_name 1～255 且响应内唯一；request_count 非负且总和不溢出 int64；success_rate/avg_latency_ms/avg_tps 为有限非负数 |

任一必需字段非法使整次响应失败，不截断字符串、不饱和数值、不部分写入。可选资源字段缺失可写 null。容量验收必须包含“单站单小时最大 flow 响应”和“单站完整用户快照”的实际序列化大小，均需低于 64 MiB；达到 80% 即触发容量评审，超过限制按 UPSTREAM_RESPONSE_TOO_LARGE 失败而不是放宽无界读取。

### 30.11 能力检查键

SiteAuthorizationResult.capabilities 只允许以下稳定 key：`status_contract`、`self_identity`、`root_identity`、`first_user_proof`、`user_pagination`、`channel_pagination`、`data_export_enabled`、`flow_contract`、`data_contract`、`flow_data_consistency`、`instance_contract`、`realtime_contract`。每项分别执行本节对应接口、HTTP/envelope、必需字段和边界校验；data_export_enabled 必须为 true。

除 flow_data_consistency 在最近 24 个完整小时均无流量时允许 skipped 外，其他项只能 passed/failed。任一 required 项 failed 时不创建历史回填；身份/首用户证明失败不保存 Token，身份通过后的接口契约/功能失败则保留加密 Token 和不可变起点，站点进入 pending_config 或 error。首次出现非空流量后立即补做 skipped 项，通过前统计状态最多为 partial，不能 ready。

## 31. 状态机和并发规则

所有站点、客户、账户状态变更先在事务中 `SELECT ... FOR UPDATE` 锁定目标行，再校验前置状态、写暂停边界和创建任务；重复提交按下文幂等规则返回当前对象或已有任务，不能依赖前端防重。

所有会访问远端的任务在创建/领取时冻结 site.config_version。任何探活、状态、元数据、事实或能力结果写库前都必须锁定 site 并再次核对版本；不一致时丢弃响应并返回 SITE_CONFIG_CHANGED。该规则覆盖 §32.1 之外的所有远端任务，防止改址、重新授权或生命周期切换期间的旧响应污染新配置。

base_url、授权凭据或 auth_status 的任何变化（含 401/decrypt_error）、required capability 可用/不可用转换（版本、data_export、必需 DTO）、disable 和 enable 共用 `BumpSiteFence` 事务：先在事务外定位受影响父任务及对象 ID，除 collection_run.site_id 和已有 run-window 外，还必须通过 target account/customer 的不可变归属识别 site_id=NULL、尚未物化子行的本地父任务。事务内按全局顺序锁 site、customer/account，再递增 config_version，随后锁 collection_run/run-window；把受影响父任务的所有未终态子行置 failed/SITE_CONFIG_CHANGED，父任务置 failed、finished_at=now、active_key=NULL 并同步对象 backfill 状态，事实 collection_window/旧事实保持不变。锁后集合变化则回滚重试。事务提交后旧在途响应因版本不一致只能丢弃。同一能力结果重复出现不再次递增。新状态允许采集时必须创建使用新 config_version 的新 run；disable/expired/capability_failed 不重排队，enable/reauthorize/recheck 在旧任务已终态且能力通过后创建恢复 run，因此快速状态切换不会重叠或复活旧请求。scope 失效事务同时把不再适用的现有告警 resolved/scope_inactive，且不得因被栅栏终止的任务新建 backfill_failed 告警。

### 31.1 站点

管理状态：

~~~text
active --disable--> disabled --enable--> active
~~~

- disable：执行 BumpSiteFence，写 `disabled_at=floorToBeijingHour(now)`，同时按 `floorToMinute(now)` 插入开放的 site_monitoring_pause，停止所有周期任务，不清除游标；
- enable：执行 BumpSiteFence，先关闭 monitoring pause 并立即恢复当前资源采集；用量 statistics_status=backfilling，从 `coalesce(last_complete_hour+3600, statistics_start_at)` 扫描到 backfillEnd，优先处理既有 missing/pending、跳过 complete/unavailable，成功后清除 disabled_at；资源停用区间保持 paused 且不做历史回填；
- auth_status 不是 authorized 或 data_export_enabled=false 时拒绝 enable，并提示先重新授权或开启数据导出；
- 有 statistics_end_at 的永久下线站点只能由 admin 先清除终止时间再恢复。

授权状态：

~~~text
unauthorized --authorize--> authorized
authorized --401/decrypt_error--> expired
expired --reauthorize--> authorized
~~~

authorize/reauthorize 成功、401/decrypt_error 进入 expired、base_url 变化都通过 BumpSiteFence 在状态事务内恰好递增一次 config_version，并按统一规则终止旧任务，从而使所有在途旧客户端结果失效。

统计状态：

~~~text
pending_config --capability_ok--> backfilling
pending_config --export_disabled--> pending_config
backfilling --all_windows_complete--> ready
backfilling --some_failed--> partial
ready --new_gap--> partial
partial --repaired--> ready
any --fatal_incompatible--> error
error/pending_config --required_capabilities_passed--> backfilling
error --identity_or_first_user_proof_failed--> error
active state --disable--> paused
paused --enable--> backfilling
~~~

online_status 和 health_status 由探活、实例采样独立驱动，不参与上述状态机。

management_status/statistics_status 的优先级固定：disabled（无论是否有 statistics_end_at）对统计显示为 paused。disabled 站点允许 authorize/reauthorize/recheck-capabilities，只更新 auth_status、site_capability 和 BumpSiteFence；成功或失败都不创建采集 run、不把 statistics_status 改为 backfilling/ready/error。失败详情由 auth_status/site_capability 展示，enable 前置检查据此拒绝；仅 active 且无 statistics_end_at 时，required capability 通过才执行 error/pending_config -> backfilling/ready。

### 31.2 账户

~~~text
create(pending backfill) --> active
active --archive--> archived
archived --restore--> archived + backfilling --success--> active
~~~

- 归档生效时间归一到当前北京时间小时起点；statistics_paused_at 为空时写入，否则保留更早值；
- archive 后当前及后续小时不生成 account_stat；所属 customer_stat 重算时排除该账户；
- restore 先保持 statistics_paused_at，重算 [statistics_paused_at, 当前最近完整小时结束) 的账户和客户统计；
- restore 创建 run 后账户保持 archived，直到重建成功才在完成事务中改为 active、清空暂停边界；失败仍为 archived；
- 所属客户仍为 disabled 时拒绝账户 restore，必须先恢复客户；
- remote_state=identity_mismatch 时拒绝账户 restore；本期不允许自动或手工恢复该绑定，只能归档旧账户，或在完全无历史且满足删除约束时删除；
- 全部成功后清空 statistics_paused_at，并将 statistics_backfill_status=none；
- 失败保持 archived 语义边界和 failed 状态，允许重试；
- 远端 missing/disabled 不自动改变 managed_status，基础事实和已有统计仍保留；identity_mismatch 会暂停账户/客户新增汇总，避免 ID 复用污染归属。

### 31.3 客户

~~~text
communicating <--> signing
communicating <--> using
signing <--> using
任一非 disabled 状态 --disable--> disabled
disabled --enable--> disabled + backfilling --success--> using
~~~

- 仅 using 客户允许创建新的纳管账户；communicating 和 signing 可保留已有账户及统计，但不能新增绑定；
- disabled 将 statistics_paused_at 归一到当前小时起点，暂停该客户及其全部账户汇总；
- disable 在同一事务内把该起点传播到全部 active 账户的 statistics_paused_at，不改变 managed_status；
- enable 的唯一目标状态为 using，并重算客户及所属非 archived、非 identity_mismatch 账户的停用区间；
- enable 创建 run 后客户保持 disabled，直到全部重建成功才在完成事务中改为 using、清空边界；失败仍为 disabled；
- 非 archived 账户重算成功后清空其 statistics_paused_at；archived 账户保留最早暂停起点，待账户自身 restore 时补齐；
- archived 账户不会因客户恢复而自动恢复。

account/customer.statistics_backfill_status 与本地重建 run 的权威转换：

~~~text
none --enqueue--> pending --lease--> running
running --success--> none
running --terminal_failure--> failed
failed --new_retry_run--> pending
running --lease_recovered--> pending
~~~

enqueue、lease、终态和租约回收都在同一事务按全局锁序锁定对象与 collection_run 后同步状态。账户任务同时影响所属 customer 时锁定 customer；每个 `(target_type,target_id,task_type)` 的当前状态只取 id 最大的最新逻辑 run，新 retry run 创建后旧 failed run 保持不可变但立即退出当前状态计算。customer.statistics_backfill_status 按客户自身及各账户各自的最新逻辑 run 聚合为 running 优先、pending 次之、failed 次之、否则 none，不扫描已被新 run 取代的历史失败记录。相同 customer 的 account/customer 本地重建使用同一 `local_rebuild:customer_id` 锁 family 串行，避免多个账户任务互相覆盖客户汇总。

所有可能同时触碰任务和统计的事务使用唯一锁序，不允许局部反转：`site（id 升序） -> customer（id 升序） -> account（id 升序） -> collection_run 父行 -> collection_run_window（hour_ts 升序） -> collection_window -> aggregation_bucket_lock（lock_key 字典序） -> fact/summary 行`。不需要的层级可跳过但不得换序。customer 跨站任务先在事务外取得不可变账户绑定的 site ID 集合，再在事务内按序锁 site、customer、account 并复核集合；不一致则回滚重试。claim、结果提交、BumpSiteFence、租约回收和状态操作全部遵守该序；死锁仍按数据库错误整笔回滚并由既有尝试预算重试，禁止在持有子行锁时反向获取父/对象锁。

### 31.4 采集任务

~~~text
pending --> running --> success
             ^      \-> pending (仍有重试预算)
             |       \-> failed (预算耗尽，终态)
             +---------------
~~~

- 同一 site_id + task_type 仅允许一个 running；
- 同一 site_id + hour_ts 的小时替换串行执行；
- 创建任务时先生成 active_key；数据库唯一键冲突时返回已有活跃任务，不重复插入；
- 同一 site_id、同一窗口型 task_type 的活动范围若完全相同则返回已有任务；若仅部分重叠，手动请求返回 TASK_OVERLAP，调度器把尚未覆盖的小时并入或另建不重叠范围，不能让两个 run 同时处理同一窗口；
- 创建任何站点窗口任务时先在事务内 `SELECT site FOR UPDATE`，再按 usage/validation 共用的窗口锁 family 查询所有 pending/running 重叠范围并插入；active_key 只负责完全同键幂等，部分重叠互斥由该行锁与范围检查保证；
- collection_run 是一个逻辑范围任务；每小时的执行状态、attempt_count、next_retry_at 和错误保存在该 run 自己的 collection_run_window，run 保存优先级、总体进度、next_attempt_at 和 heartbeat_at；
- 触发对象状态与 collection_run 父行必须在同一事务提交；窗口型父行先以 windows_initialized_at=null 持久化，再由 §32.9 的通用幂等物化器创建目标 run-window，全部核对后才可领取。新 retry run 拥有自己的 run-window 并从 0 使用预算；同一 run 崩溃恢复不重置，旧 run-window 永不转移或覆盖，因此小时采集、回填、校验和人工补采的历史与预算彼此独立；
- usage_validation 对已 complete 事实窗口创建 pending run-window，但不修改 collection_window.status；只有验证发现不一致时才把事实窗口改为 missing 并隔离旧事实。run 的 progress/success/failed 只读取 run-window，不能把事实 complete 当作该 validation run 已执行；
- 父 run 领取及窗口 claim 均先按全局锁序锁定相关 site/customer/account，再锁 collection_run 父行；仅当 windows_initialized_at 非空时，才以 `status=pending AND coalesce(next_retry_at,0)<=now`、hour_ts 顺序 `SELECT collection_run_window ... FOR UPDATE SKIP LOCKED`。同一事务把选中子行置 running、`attempt_count=attempt_count+1`、写 started_at，并生成本次 attempt/CAS token 写入父行 last_request_id、刷新 heartbeat。后续心跳和结果提交必须同时匹配 run_id、running 状态和该 last_request_id；远端请求必须在提交该领取事务后执行，任何路径都不能直接处理未领取的子行；
- 实时小时任务优先级高于历史回填；
- Worker 每次领取最多处理 24 个小时或 30 秒，然后更新进度并重新入队；
- 范围任务的单个 run-window 失败不会中止其他窗口；未到 next_retry_at 的行暂时跳过。全部 run-window terminal 后，failed_windows=0 才标记 run success，否则标记 failed；事实窗口状态另行决定站点 ready/partial；
- 尚有可重试窗口但当前均未到时间时，run 置回 pending，next_attempt_at 取最早 next_retry_at，避免忙轮询；
- 单实例进程启动时无条件接管全部 running 父任务；运行期间每 60 秒回收 heartbeat_at 超过 5 分钟的 running 租约。接管/回收事务先按全局锁序锁 site/customer/account，再锁父 run 及其 running run-window；领取时已经预留的尝试不重复计数。窗口型 run 仅按各子行 attempt_count 判断预算：有预算的子行置 pending 并令 next_retry_at=now，耗尽的置 failed 并写 WORKER_LEASE_LOST，随后重算父计数/status/next_attempt_at，父 retry_count 不递增也不参与预算。非窗口 run 没有 run-window，仅按领取时已增加的父 retry_count 判断回 pending 或 failed；
- account/customer 本地重建父 run 被接管或回收时，在上述同一事务按既定对象优先锁序重算 statistics_backfill_status，不能出现父任务已 pending 而对象永久 running；
- 窗口型任务的 collection_run_window.attempt_count 达上限后保持 failed；非窗口任务的 collection_run.retry_count 达上限后保持 failed。两者都由告警暴露且不丢弃任务范围，禁止混用两个计数器。
- failed 是逻辑 run 终态，不再把同一 run 改回 pending；admin 重试会创建新 run 和新的 run-window，不改旧任务历史。

collection_run_window 状态为 pending/running/success/failed/unavailable。可重试错误在预算内回到 pending 并设置 next_retry_at，预算耗尽才 failed；unavailable 是已执行的 terminal，但不把事实窗口伪装为 complete。collection_run.completed_windows 统计 success，failed_windows 统计 failed，progress 分子为 success+failed+unavailable。

collection_window：

~~~text
pending --> complete
pending/complete --> missing
missing --> complete
pending/missing --> unavailable
~~~

paused/backfilling 是查询结合站点 lifecycle 和活动任务推导的 DataStatus，不是 collection_window 持久状态；collection_window 只保存 pending/complete/missing/unavailable。

unavailable 只允许在上游返回可机器识别的“该历史范围不可提供”契约时设置；当前参考 new-api 未提供该错误码，因此首版适配器默认不会主动产生 unavailable。网络错误、空数组和人工判断均不得设置为 unavailable。

### 31.5 导出

~~~text
pending --> running --> success --> expired
             ^      \-> pending (可重试且 attempt_count<2)
             |       \-> failed (预算耗尽或不可重试)
             +---------------
~~~

第一次可重试失败删除半成品、保留 active_key，设置 pending/next_attempt_at；第二次或不可重试失败才置 failed 并清 active_key。失败终态可由用户点击“重新导出”创建新任务，不复用原任务 ID。文件过期后只删除文件并改 expired，保留任务元数据。

### 31.6 告警

~~~text
不存在 --首次命中--> pending --达到 for_times--> firing
pending --条件恢复--> resolved
firing --条件恢复--> resolved
resolved --再次命中--> 新建事件
~~~

pending 不发送通知；firing 和 resolved 各最多创建一条钉钉投递。

每次已提交业务样本用 canonical `(active_key, observed_at, sample_key)` 驱动状态机。alert_evaluation_cursor 与事件变化在同一事务推进：同一 sample 幂等、更早 sample 忽略、同一 observed_at 的不同 sample_key 拒绝为冲突，unknown 不推进。源事务提交后读取精确 committed snapshot；hook 失败由每 5 分钟使用同一 evaluator 的扫描收敛，不能在提交前投递或另写一套兜底状态机。

## 32. 采集和统计事务

### 32.1 小时窗口替换

远端请求在数据库事务外执行，只有完整响应通过校验后才能进入替换事务。

~~~text
CollectSiteHour(runID, siteID, H):
  assert H is aligned to 3600 and H+3600 <= lastCompleteHourEnd
  acquire process lock("usage:site:" + siteID + ":hour:" + H)
  assert collection_run(runID) is running and covers [H,H+3600)
  flowRows = GET /api/data/flow?start=H&end=H+3599
  dataSummary = GET /api/data?start=H&end=H+3599
  validate both success, nonnegative metrics, bounded strings
  facts = group flowRows by (user_id, original model_name, channel_id)
  compare facts aggregated by original model_name with dataSummary
  source_hash = SHA256(canonical sorted facts)

  begin transaction:
    select site/site-related customer/account rows in global lock order
    select collection_run(runID) for update and assert status=running
    assert site.config_version == collection_run.site_config_version
    assert current base_url/auth/management state still permits this run
    select collection_run_window(runID,siteID,H) for update and assert status=running
    select collection_window(siteID,H) for update
    for lockKey in lexicalSort(site-hour,site-date,global-hour,global-date):
      insert ignore aggregation_bucket_lock(lockKey)
      select aggregation_bucket_lock(lockKey) for update
    delete usage_fact_hourly where site_id=siteID and hour_ts=H
    bulk insert facts in batches of 500
    upsert collection_window(status=complete, fetched_rows, source_hash, last_fact_run_id=runID)
    rebuild site/model/channel hourly for siteID,H using complete windows only
    rebuild account/customer hourly for siteID,H with current pause boundaries
    rebuild global hourly for H from facts whose site window is complete
    rebuild usage_fact_daily and all six daily tables for Beijing date(H) from complete windows only
    advance cursor only through the highest contiguous complete hour
    mark collection_run_window success with fetched/written counts
    recalculate collection_run window counters and heartbeat
  commit

  if all run windows in this run are terminal:
    mark success when failed_windows=0, otherwise mark failed
  else:
    return run to pending with the earliest retry/continuation time
~~~

如果 facts 为空，事务仍执行删除、窗口 complete、汇总重建和游标推进，从而正确表达零流量。事务失败时旧事实和旧汇总完整回滚。

aggregation_bucket_lock 的动态行创建本身也会加锁，因此所有小时采集和本地重建都必须按 lock_key 字典序逐个执行 `INSERT IGNORE` 后立即 `SELECT ... FOR UPDATE`；禁止先按业务顺序批量 upsert、再排序 SELECT。

远端/校验失败先更新当前 run-window 的重试状态。usage_hour/backfill 在事实从未 complete 时同步把 collection_window 标为 missing；重拉既有 complete 窗口发生网络失败时保留旧 complete。usage_validation 的传输/5xx 失败只使 run-window 重试并保持事实 complete、daily is_final=false；预算耗尽进入 failed 时以 failure_kind=execution_failed 触发该小时 validation_failed 告警。只有两个上游接口明确返回且相互不一致时才把事实窗口降为 missing，并以 failure_kind=data_mismatch 触发同一逻辑规则。

远端请求在事务外执行，因此 config_version 栅栏是强制步骤。地址、授权或生命周期在请求期间变化时返回 SITE_CONFIG_CHANGED 并丢弃旧响应；只有新状态仍允许采集时才由 BumpSiteFence 后续流程创建新 run，disabled/expired 不重排队。不得把旧主机数据写入原 site。aggregation_bucket_lock 使小时采集与本地账户/客户重建在相同 site/hour/date 上互斥，并使跨站 global 重建串行；锁顺序固定以避免死锁。

group 时 model_name 不做别名或 trim；同一键出现多个 username 时，username_snapshot 选择非空用户名按字典序最小值，保证重复采集结果确定，账户绑定和活跃去重始终只使用 remote_user_id。

### 32.2 小时汇总 SQL 口径

站点：

~~~sql
SELECT SUM(request_count), SUM(quota), SUM(token_used),
       COUNT(DISTINCT remote_user_id)
FROM usage_fact_hourly f
JOIN collection_window w ON w.site_id=f.site_id AND w.hour_ts=f.hour_ts AND w.status='complete'
WHERE f.site_id = ? AND f.hour_ts = ?;
~~~

模型在上式增加 GROUP BY model_name；通道增加 GROUP BY channel_id。

账户通过 account.site_id + account.remote_user_id 关联事实，并继续要求对应 collection_window=complete。目标小时还必须满足 `remote_created_at < H+3600`；用户注册整点之前不属于账户 expected 范围。历史有效性只由时间边界判断：`statistics_paused_at IS NULL OR H < statistics_paused_at` 时该小时可生成。当前 managed_status/customer.status 只负责建立或清除暂停边界，不能在重建旧日期时删除暂停边界之前的合法历史。

客户按 customer_id + site_id 分组，并从当时已注册且目标小时早于各自暂停边界的账户及 complete 窗口关联事实；仅当该站该小时至少存在一个满足 remote_created_at 与历史暂停边界的账户时才属于客户 expected 范围。active_users 使用 COUNT(DISTINCT account.id)。客户当前为 disabled 但 H 早于 customer.statistics_paused_at 时仍保留历史；H 位于该边界及之后才返回 paused。

客户在目标小时存在至少一个历史有效账户时，仅以这些账户作为当时口径；当前 archived/identity_mismatch 但 H 早于其暂停边界的账户仍计入，H 位于边界及之后才排除。若 H 位于客户暂停边界，或该小时所有已注册账户均已进入各自暂停边界，则客户桶返回 null + paused，而不是 complete 0。

全局仅从 expected site 中 collection_window=complete 的事实按 hour_ts 汇总，active_users 使用 COUNT(DISTINCT CONCAT(site_id, ':', remote_user_id))。如果 expected 窗口不是全部 complete：

- 至少一个 complete：保存已知合计，data_status=partial；
- 没有 complete 且存在 missing/pending：不保存数值行，查询返回 null；
- 全部 complete 且无事实：删除稀疏汇总行，查询补 0。

### 32.3 expected site 判定

站点对窗口 [H,H+3600) 属于 expected，当且仅当：

~~~text
statistics_start_at < H+3600
AND (statistics_end_at IS NULL OR statistics_end_at > H)
~~~

management_status=disabled 但 statistics_end_at 为空时，该窗口状态为 paused，仍计入完整率分母；恢复并补齐后转 complete。永久下线后的窗口不进入分母。

站点停用期间不预先为未来小时批量插入 collection_window；查询时根据 disabled_at、management_status 和恢复任务状态动态推导 paused/backfilling。恢复任务从 `coalesce(last_complete_hour+3600, statistics_start_at)` 扫描到 backfillEnd，先处理更早的 missing/pending，再为停用区间未记录小时创建 collection_run_window=pending，并仅在对应事实从未 complete 时 upsert collection_window=pending，跳过 complete/unavailable；全部可恢复窗口成功后才清空 disabled_at。账户和客户的 paused 状态同理由各自 statistics_paused_at 推导，不写入站点 collection_window。

### 32.4 游标推进

last_complete_hour 只能指向从 statistics_start_at 起连续 complete 的最高小时。不能因为未来某小时成功而跨过 missing 窗口。查询下一批回填范围时：

1. 优先扫描 collection_window 的 missing/pending；
2. 再从 last_complete_hour+3600 向后扫描，跳过已有 complete/unavailable 的窗口，只为未记录或可重试窗口建任务；
3. unavailable 仍会阻止 last_complete_hour 跨过该小时；后续调度通过 collection_window 跳过已确认 unavailable 的窗口并继续处理更晚窗口，站点 statistics_status 保持 partial。

任何历史校验把 complete 改为 missing/unavailable 时，必须在同一业务操作中回退 cursor 到该窗口之前的最高连续 complete 小时，删除该小时的六级可见汇总并重建对应 daily/global。旧事实保留但因窗口非 complete 不参与查询。

### 32.5 daily 重建

RebuildDate(date_key) 从北京时间当日 00:00 到次日 00:00 的 `usage_fact_hourly JOIN collection_window(status=complete)` 重建：

- usage_fact_daily 按 site、remote_user、model、channel 聚合；
- 六级 daily 使用与 hourly 相同的业务范围；
- daily active_users 在整日范围重新 COUNT DISTINCT；
- 当天未结束时 is_final=0，as_of 为最近 complete 小时结束；
- 次日校验 24 个窗口全部通过后 is_final=1；
- expected 窗口全部 complete 时为 complete；部分 complete、部分非 complete 时统一为 partial；没有任何 complete 时再按 missing > unavailable > paused > pending/backfilling 选择状态；
- daily 完整零流量不落统计行，完整状态从 collection_window 推导。

重建采用“删除目标日期相关行 + 批量插入新聚合”的单事务方式，不做旧值累加。

### 32.6 月、年和任意区间

- request_count、quota、token_used 从 daily SUM；
- active_users 不从汇总表相加：小时和未结束当日使用 usage_fact_hourly，完整日/月/年优先使用保留相同身份维度的 usage_fact_daily 重新去重；
- 账户/客户查询先关联 account 固定映射；
- 模型/通道查询覆盖站点全部用户；
- 月、年查询不写缓存表，必要时可使用短期进程内查询缓存，缓存键必须包含角色无关的全部筛选参数和最后数据版本号；
- 任意跨站金额由 site_breakdown 的 quota 分项计算，global_stat.quota 不用于金额。

### 32.7 次日与每周验证

每个待验证小时：

1. 获取 /api/data/flow；
2. 获取 /api/data；
3. 比较按模型汇总的三个指标；
4. 计算 source_hash 与本地窗口哈希；
5. 完全一致时只更新 verified_at；
6. 远端两接口一致但 source_hash 与本地不同，执行完整窗口替换；
7. 远端两接口自身不一致时不覆盖旧事实，在事务内把窗口标记 missing、隔离旧事实、回退游标、重建汇总并告警；
8. 日期内属于各站统计范围的 expected 小时全部通过后置 is_final=1；普通完整日为 24 小时，首日和永久终止日只校验范围内窗口。

每周任务处理最近 7 个已结束自然日；手动补采可以指定更早范围。

### 32.8 新站点历史回填

1. `/api/user/self` 身份校验后精确读取候选 root，并完成一次包含软删除记录的完整用户快照；
2. 校验候选 root 为快照最小正 ID、created_at 不晚于任一用户，再将已验证 Token 与 root_created_at 原值、`statistics_start_at=floorToBeijingHour(root.created_at)`、`statistics_start_source=root_created_at` 原子保存；之后起点不可修改，重新授权必须重复证明并核对 ID/时间原值一致；
3. 完成 enable_data_export、必要接口和 flow/data 能力检查；步骤 1 的快照作为首次用户同步结果，start=statistics_start_at；
4. 创建一个覆盖完整范围的 collection_run，并为每个待处理小时创建 collection_run_window=pending；仅当事实窗口不存在或非 complete 时 upsert collection_window=pending，绝不因任务 pending 隐藏已有 complete 事实；范围任务以 24 窗口或 30 秒为执行片，优先级低于实时小时任务；
5. 单站内顺序执行，站点间最多 2 并发；
6. 每个小时同时请求 /api/data/flow 和 /api/data，两个接口按模型汇总一致后才允许写入 complete 窗口；
7. 每处理完一个小时立即可查询部分数据；
8. 进度 = terminal_window_count / expected_window_count，并同步写入 run 的 total/completed/failed 窗口计数；
9. unavailable 计入 terminal，但不计 complete；
10. 所有可恢复窗口 complete 且无 unavailable 时 ready，否则 partial。

若 root.created_at 位于当前未结束小时，expected_window_count=0，初始化任务直接 success、进度记为 1，站点可进入 ready；首个完整小时仍由按 usage_delay_minutes 调度的正常任务采集，不能除以 0 或长期停在 backfilling。

读取不到合法 root、完整用户快照为空或首用户证明失败时 statistics_status=error，接入终止且查询不得返回“完整零值”。空快照一律返回 SITE_INCOMPATIBLE，不存在“仅凭精确 root 读取继续”的例外。

### 32.9 新账户历史生成

账户创建必须与 durable 父任务原子提交：精确读取远端用户在事务外完成，随后事务按全局锁序复核 site/customer、插入 account（statistics_backfill_status=pending）、插入冻结 start/end 范围的 account_rebuild collection_run（windows_initialized_at=null），并按最新 run 聚合规则把 customer.backfill_status 更新为 pending。任一步失败全部回滚，不存在已提交账户却没有 run 的断点。事务后窗口物化器按父 run 不可变范围幂等创建 run-window：

~~~text
for each date in [floorToBeijingHour(remote_created_at), latest complete hour] that is expected for the account:
  rebuild account daily and underlying account hourly
  rebuild customer hourly/daily for affected customer and dates
~~~

物化器只扫描 `status=pending AND windows_initialized_at IS NULL`，只使用父 run 已冻结的 start_timestamp/end_timestamp 和创建时 scope，不随当前时间扩张。每个最多 1,000 行的批次都按全局锁序锁 scope 对象→父 run，并在锁内再次断言 parent 仍 pending/init null；站点相关任务还必须核对冻结 config_version 等于当前 site 且 lifecycle/capability 允许。断言失败立即停止，不插入子行。依靠 `(run_id,site_id,hour_ts)` 唯一键支持崩溃续作。最终事务同样用上述 CAS 条件，核对实际行数等于 expected_window_count 后才写 total_windows/windows_initialized_at；expected_window_count=0 也只有 CAS 成功才可把 run 置 success并重算对象状态。BumpSiteFence 并发把 parent 置 failed 后，补偿器不得新增 pending 子行或改写终态。启动和每分钟只补偿仍 pending/init null 的窗口 run；新产生小时由后续正常调度或新恢复 run 处理。

本地重建不调用远端。只检查账户注册后 expected 范围内的 collection_window；存在 missing 时先创建站点窗口补采依赖，账户任务状态保持 pending；依赖完成后继续。注册前的站点缺口与该账户无关。

账户本地重建按 site/date 分片，并获取与小时采集相同的 `stats:site:*` 桶锁后重建 account/customer；客户跨站恢复也拆成按 site、date 的确定顺序子窗口，不能在一个无序大事务中同时锁多个站点。新账户首次回填的成功子窗口逐步可见；账户 restore/客户 enable 的恢复区间继续由 statistics_paused_at 隔离，全部子窗口成功后才在对象完成事务中清除暂停边界、切换 active/using 并原子开放，失败时不得暴露部分恢复区间。

若该远端用户在完整站点窗口内从未产生事实，账户本地回填仍应 success；查询按站点窗口完整性返回 complete 零值，不能因为没有 account_stat 行长期保持 pending。

### 32.10 暂停和恢复边界

- 暂停生效时间统一 floorToHour(now, Asia/Shanghai)；
- identity_mismatch 写入暂停边界时与归档使用相同的汇总删除事务，但不改变 managed_status；
- 暂停时删除该边界及之后已提前生成的账户/客户 hourly 和受影响 daily，避免当前小时混入；
- 恢复时 backfillEnd=最近完整小时结束；
- 账户恢复重建自身及所属客户；
- 客户恢复重建客户及全部非 archived、非 identity_mismatch 账户，并只为这些成功重建的账户清空 statistics_paused_at；
- 账户 restore 成功后清空该账户 statistics_paused_at；客户 enable 成功后清空客户及已成功重建的非 archived 账户 statistics_paused_at；
- 恢复发生在当前未结束小时内时，该小时在结束后按 active 状态正常生成。

### 32.11 时间轴补齐

统计查询先生成目标粒度时间桶，再左连接统计结果与 collection_window：

| 来源状态 | 返回 value | data_status |
|---|---|---|
| 全部 complete 且有行 | 精确值 | complete |
| 全部 complete 且无行 | 0 | complete |
| 部分 complete | 已知合计 | partial |
| 无 complete，存在 missing/pending | null | missing/pending |
| 上游不可提供 | null | unavailable |
| 对象暂停 | null | paused |

API 同时返回 is_partial、complete_site_count、expected_site_count、missing_site_ids 和 missing_ranges。前端不得自行把 null 转 0。

missing_ranges 将同一 site_id、同一状态的相邻小时合并为 [start_timestamp,end_timestamp)；按开始时间倒序最多返回 100 段，并返回 missing_range_total 和 missing_ranges_truncated，避免多年缺口撑大响应。

expected 的已结束小时如果没有 collection_window：被活动 run 覆盖时推导为 pending；站点 disabled 且无 statistics_end_at 时推导为 paused；其余超过 `hour_end + max(15, usage_delay_minutes+10) 分钟` 仍无窗口时推导为 missing。默认 `usage_delay_minutes=5` 时截止点为 H+15；调整延迟后按同一公式动态变化，不会在任务计划启动前误报 missing。当前未结束小时不进入业务统计范围。该推导规则保证任务丢失也不会被误判为 complete 零值。

### 32.12 用户和通道同步

用户同步每站每小时执行一次完整分页：

1. GET /api/user/ 读取全部当前用户；
2. 只有全部分页成功、累计数量与响应 total 一致且未超过页数保护上限时，才将本轮标记为完整快照；
3. 分页中途失败、total 异常变化或结果重复冲突时，本轮整体失败，不更新任何账户为 missing；
4. 建立 remote_user_id → RemoteUserItem 映射，并保留上游 `DeletedAt.Valid`；软删除项参与首用户证明但在账户存在性判断中等同于未出现；
5. 只对未软删除的已纳管用户更新 account 的 username、display_name、group、status、quota、used_quota、request_count；created_at 与已存 remote_created_at 不一致时不覆盖身份和属性，设置 remote_state=identity_mismatch、statistics_paused_at=当前整点并触发 account_identity_mismatch 告警；
6. 已纳管 ID 在完整快照中缺失或对应项已软删除时 remote_missing_count+1；达到 2 次才设置 remote_state=missing，单次缺失只保留计数；
7. missing 账户再次出现且 created_at 与原值一致时立即恢复 normal、remote_missing_count=0，并更新 last_remote_seen_at；identity_mismatch 不自动恢复，避免冲突期间事实误归属；
8. 远端用户名变更只更新快照，不改变绑定；
9. 同一事务完成账户快照更新，事务后批量评估账户告警；remote_status=1 为启用，其他值命中 account_disabled 规则。

通道同步每站每小时执行一次：

1. GET /api/channel/ 完整分页；
2. 仅在全部分页成功且结果被判定为完整快照后进入更新事务；
3. 以 site_id + remote_channel_id upsert name 和 last_synced_at；
4. 本轮出现的通道 remote_missing=0；
5. 只有完整快照中未出现的旧通道才设置 remote_missing=1；失败或不完整快照不修改 missing 状态；
6. 不删除旧通道，不读取 key、余额或健康；
7. channel_id=0 使用固定显示“未知通道”，不插入 site_channel。

### 32.13 当前状态和资源采集

探活：

- GET /api/status 网络和 HTTP 成功后写 online、last_probe_at、last_probe_success_at 并清零 probe_fail_count；只有 envelope 和字段契约通过时才同时更新版本、系统名、有效汇率和导出开关；
- 失败写 last_probe_at、probe_fail_count+1；达到 3 次才写 offline；
- data_export_enabled 从 true 变 false、必需 DTO 从兼容变不兼容时执行 BumpSiteFence，statistics_status=pending_config/error，终止旧用量任务并触发对应告警；同一失败状态的后续探活只更新 checked_at，不重复 bump。修复后必须由 recheck-capabilities 执行反向 capability 转换并创建恢复 run：有游标从缺口补齐，无游标从 statistics_start_at 创建首次全历史回填。

RPM/TPM：

- GET /api/log/stat 成功后覆盖 site.current_rpm/current_tpm/last_realtime_stat_at；
- 失败不把当前值清零，只让 freshness 过期；
- 不写任何历史表。

资源：

- GET /api/system-info/instances 成功后按 node_name upsert site_instance，同时保存上游原始 status/stale_after_seconds；
- 首次能力通过时设置不可回退的 site.monitoring_start_at=floorToMinute(now)；资源查询不把平台接入前的历史列为缺口；
- 将当前秒归一到 minute_ts，同一分钟重复采样使用 upsert 覆盖；每个成功分钟必须写 site_status_minutely，即使实例数为 0；
- effective stale 使用实际生效的 `instance_stale` 规则阈值。已知实例连续 3 个成功快照未处于 online 后 effective_status=offline；达到 stale 阈值但未满 3 次时为 stale；接口失败为 unknown，不增加或清零该连续次数；
- 上游本轮未返回、但上次存在的实例也写该分钟的 stale/offline 行，资源指标为 null，不立即删除；
- 实例 expected 从 first_seen_at 开始，到 retired_at 前结束。一次成功且完整校验通过的实例快照是当前节点目录：此前 active、但本次未出现的 node 在同一提交中写 retired_at，不再合成 stale/offline 样本、不计 expected，并以 resolution_reason=retired 结束该 node 的活动告警。上游请求失败、非法响应或 unknown 轮次不得退役任何 node；已退役 node 重新出现时清除 retired_at，并从该明确样本重新进入活动范围；历史实例行、资源样本和已结束告警保留；
- 生成 site_status_minutely 时对本轮实例直接聚合，不按 hostname 去重；
- 每个完整小时结束后从 minute 表重建实例和站点 hourly，并固化 sample_count、expected_sample_count、data_status；
- 当前日每小时同步重建 daily，次日 03:00 final；
- 整个实例接口失败时不写虚假离线分钟样本，只记录采集失败并由站点/实例告警状态表达未知。

expected_sample_count 从 monitoring_start_at、site_monitoring_pause 和 statistics_end_at 计算；成功快照数为 sample_count。站点恢复后 pause 历史不删除，因此分钟清理后 hourly/daily 仍能区分 intentional paused 和异常 missing。

Scheduler 启动、每个整点和每日 03:00 都扫描“存在分钟数据或 expected 区间但缺少/未 final 汇总”的资源 hour/day 并幂等重建；进程跨整点宕机不会永久漏聚合。分钟清理事务只删除目标小时/日期的实例和站点汇总均已存在且 sample/expected/data_status 已固化的数据，否则跳过并告警，不能先删后失去重建依据。

# 33. 上游日志明细采集

官方基线为 `7c28993f6bd9e92616f3f578212577f8b7c40b45`。`GET /api/log/` 仅在授权且上游日志能力开启时调用；使用 `p/page_size` 分页，按时间窗口重叠拉取，使用稳定键去重，禁止使用 ClickHouse 展示 ID 游标。每页请求带 request ID、固定 request timeout 和最大响应体限制。

日志 run 在事务外请求，写入前锁定并核对 site/config_version；旧版本响应以 `SITE_CONFIG_CHANGED` 丢弃。成功完整窗口才更新 completeness cursor，分页失败保留上一轮事实。`type`、时间、用户名、模型、token、channel、group、request_id、upstream_request_id 全部保留为可筛选字段；IP 最小安全默认不保存。内容字段执行凭据脱敏并限制长度。

日志事实独立于 usage_fact，不写入 quota/request 汇总。每日按 `logs.retention_days` 批量清理，清理失败可重试且不影响业务采集。
# 34. 全站用户库存快照

`user_sync` 调用 pinned `GET /api/user/`，每页固定 100，校验 page/page_size/total 和累计响应上限。先在内存完成全部页、ID 唯一、created_at/字段边界和 100000 行上限校验，再进入一个事务锁 site、库存行和 account 行；config_version/URL/root/token/lifecycle 任一变化返回 SITE_CONFIG_CHANGED。

事务按 remote_user_id 顺序处理：当前项 upsert；DeletedAt 标记 deleted；完整页未出现的旧行 missing_count+1 并立即标记 missing；created_at 冲突标记 identity_mismatch 且不覆盖旧身份指标。随后用同一观察集更新已纳管 account，最后替换该小时 role/status/group 汇总。任一步失败整轮回滚。
# P0-D 官方渠道契约与状态机

官方契约固定为提交 `7c28993f6bd9e92616f3f578212577f8b7c40b45`：管理员 `GET /api/channel/`，请求 `p/page_size=100`，响应 envelope data 包含 `page/page_size/total/items`。官方 controller 使用 GORM `Omit("key")`，但平台仍以严格安全 wire DTO 防御，只声明允许字段并忽略/拒绝敏感字段漂移策略中的 Key 读取可能。

分页必须验证 page/page_size/total 稳定、单页和跨页 ID 唯一、最终数量等于 total，并设 100000 条与总响应体上限。channel_sync 获取完整快照后进入单个事务：锁定 site、校验 base URL/credential/config_version/auth/runnable fence、锁定旧渠道、upsert 当前项、标记缺席项 missing、替换当小时 complete 聚合。任一步失败回滚全部写入；失败不得刷新 synced_at、missing_count 或历史。
# P0-E 性能采集状态机

`performance_sync` 先请求 summary 枚举 model，再逐 model 请求标准明细；严格验证 series_schema、group 唯一、bucket 对齐、范围、decimal 有限性和跨 group/model 重复。标准 pinned 响应识别为 `average_only` 并原子替换所请求窗口的 official_average 行；若未来响应明确包含完整 counters，则写 counter_ready。部分 model 失败、配置版本变化、分页/响应异常或超时均回滚本轮全部事实。

标准 endpoint 缺失/403/404 记 capability unavailable，不清除旧历史；average_only 是成功能力而不是失败。所有外部请求携带 request ID 与 bounded timeout。配置栅栏与凭据轮换后旧响应不可提交。

# F1 上游完整快照

`topup_sync` 与 `redemption_sync` 独立分页读取管理员接口，每页固定 100。第一页冻结 total 与 max_id，采集完成后重新读取第一页验证 total/max_id 未变化；中途任何 total/max_id 漂移、重复 ID、非严格降序、空中间页、非法 raw number、敏感字段进入平台 DTO 或超过 100000 行均失败。只有完整验证后才在一个事务内校验 site base URL/auth/root token/config_version/management fence，upsert 当前行并推进 missing。

两任务拥有独立 active_key、lease、重试与周期调度；任一失败不得改写上次成功事实或误标 missing。请求携带派生 request ID 和 bounded timeout；错误、日志和任务 params 只允许安全错误码与 site_id，不回显原始上游对象。

# T1 任务采集状态机

`upstream_task_sync` 每轮完整分页最近 48 小时窗口，并对数据库中所有非 SUCCESS/FAILURE 的 task_id 发起精确查询；分页执行 total/maxID 末次 fence、100000 上限、重复/乱序校验。合并结果按 remote id 去重，updated_at 不能回退。config fence、独立 lease、三次重试和事务 upsert 与其他 metadata task 相同。成功提交后仅清理 retention cutoff 前终态，非终态永不清理。
管理员模型目录契约为 `GET /api/models/` 分页响应；未知字段先丢弃后严格解码，pricing、billing_expr、endpoints、bound_channels 等字段永不进入内部 DTO、日志或存储。`model_meta_sync` 遵循租约、最多三次尝试、配置版本 fence；channel snapshot 非完整时不得重建模型映射。
`plan_sync` 使用管理员无分页计划列表，进行 body/hard-cap/duplicate 校验，并在站点配置锁内原子提交；失败不制造 missing。
# D138 pricing/group collector

`pricing_group_sync` 是可恢复持久任务，冻结 `site_id/config_version` 后分别请求 `/api/group/` 与 `/api/pricing`，请求携带 request ID、root Access Token 和有界超时。decoder 只声明 D138 安全 DTO；decimal 先按 raw token 严格解析再规范化为文本，数组排序去重，未知字段忽略但未知结构、重复 identity、空组名、非法 decimal、body 超限均令对应资源整轮失败。

提交时锁定 site 并复核 config fence。group 与 pricing 分资源原子提交和维护 completeness：一个资源成功不依赖另一个资源成功，但失败资源不得 partial upsert、不得推进 last_complete_at 或 missing。重试遵循持久任务退避；disabled/unauthorized/required capability 失败时保持已知目录并记录 paused/unavailable。

能力检查只确认两个只读端点的可用性和安全响应结构。不得调用 Token 库存、`/api/performance/stats`、affinity、Codex usage 或任何远端 mutation 作为 D138 的 fallback/enrichment；这些端点的不可聚合或有副作用边界以 D138 为权威。

# D139 system-task collector 与状态机

`system_task_sync` 按站点冻结 config_version，先请求 Root `GET /api/system-task/list?limit=100`，再固定请求五个 `GET /api/system-task/current?type=<allowlist>`。每个请求携带 request ID 和有界超时。严禁 `GET /api/system-task/:task_id` 逐行补详情；current 只补 active，不能扩张为 N+1。decoder 使用 type 判别 typed 白名单，原始 JSON 与错误原文不进入中间 DTO。

list 恰好 100 行即 `truncated=true/data_status=partial`；ID 重复、非严格倒序或相邻 gap 置 `id_gap=true/partial`。非法行、未知 type/status、typed 数值溢出或结构不匹配使相应 resource 失败并记录稳定本地 code，不保存原文。五个 current resource 独立记录成功/失败；current 返回 nil 表示该 type 当前无 active，返回 pending/running 时按 remote ID 与 list 合并，返回终态或 type 不一致视为无效响应。只有 list 未截断、无 gap 且五个 current 均成功，站点本轮才可 complete。

提交事务再次核对 config_version，并原子 upsert 安全任务行与六个 resource state。失败保留上一轮任务事实和最后成功 as_of，不把未出现项标 missing；运行中断后以 durable state 和 active `system_task_sync` lease/fence 恢复。清理 worker 每日只删除早于 retention cutoff 的 succeeded/failed，pending/running 无条件保留；后续观察到终态后才开始按该终态 updated_at 计算 retention。

# D140 调度资格与生命周期触发

单站点任务全集、分类和触发类别以 02 §7.1.1/F12 为权威。Scheduler 只负责按功能周期产生候选任务，Repository/Worker 在领取与提交时继续复核 lifecycle、授权、capability、data_export 和 config_version fence。offline 站点只允许 `site_probe`；非窗口目录任务不因 data_export=false 被错误阻断，usage 三任务必须要求该开关。

创建站点和地址 preflight 不产生 collection_run。授权 ready 后创建初始 `usage_backfill`，事务提交后触发 immediate probe；其他授权后即时同步要求继续由各专项决策定义，D140 不删除或弱化。enable 在同一生命周期切换中创建恢复 backfill，并为 probe/realtime/resource 创建可恢复持久任务，使当前状态立即恢复且保留可查诊断。任何重复提交必须复用 active run 或明确返回未创建，不能用内存调用冒充 durable recovery。

# D141 授权后目录同步与维护状态机

授权事务在身份、Token、root 起点和 required capability ready 状态提交成功后，由 post-commit coordinator 以新 `site.config_version` 创建 `pricing_group_sync`。enqueue 使用规范 active_key 幂等；响应前创建失败或 post-commit 崩溃由持久补偿扫描重试。失败不能回滚或降级已提交授权，只写 `site_id/config_version/request_id` 与稳定本地 code 的安全诊断，禁止记录 Token、上游响应或错误原文。任务领取和提交继续复核 authorized/active/capability/config fence，旧版本任务以 SITE_CONFIG_CHANGED 终止。

resource daily finalize 与 gap repair 使用北京时间自然日、可控 Clock、持久日水位和有界游标。daily 只处理前一完整日；hourly/daily repair 只处理存在 expected 或分钟事实但聚合缺失/未固化的桶。所有 scope 成功前水位保持未完成，部分失败不能把日期、站点或桶标 complete/final。collection_run error redaction 和成功 metadata 诊断 run cleanup 同样按可控 Clock cutoff、稳定 id 游标和有界批次执行；删除/清空谓词必须在事务内再次核对，防止扫描后状态变化误删。

实例目录成功快照必须同步维护 `site_instance_lifecycle`：首次/重现打开 known 区间，缺失退休关闭 open 区间，同分钟退休重现合并，数据库唯一 open key 防止并发产生双 open。迁移回填的 legacy_unknown 在首次新样本边界切断，维护任务只消费 known 证据。gap repair 按 `(resource_kind,site_id,node_name,bucket_start)` 稳定复合游标、F13 批次 2 逐桶恢复，只修 missing/unfixed hourly；已正确 complete 桶的所有字段和 `last_calculated_at` 不变，daily/final 不归 gap 改写。daily finalize 锁 site/config/lifecycle revision，在同一 fenced scope 由完整 expected hourly 集合重建 instance/site daily，校验四态、聚合守恒和两类 daily 都存在后才 final。

自动 export/log/system-task retention 继续按各自 TTL/终态规则最终收敛，但其具体墙钟相位不属于 D141。完整 operation_id、默认调度、谓词和完成条件以 05C §34.7/F13 为权威。
